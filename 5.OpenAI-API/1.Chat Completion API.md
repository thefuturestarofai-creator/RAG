# Chat Completion API

## 什么是 Chat Completion API？

想象你有一个**万能遥控器**（Chat Completion API），你可以对着它说话（发送消息），它就能控制电视（大语言模型）给你播放想要的内容。

Chat Completion API 是 OpenAI 提供的最核心的接口，它让开发者可以通过代码与 GPT 等大语言模型进行对话。

## 核心概念

### 1. 三种角色（Role）

就像一场三人对话：

| 角色 | 英文 | 作用 | 类比 |
|------|------|------|------|
| 系统 | `system` | 设定 AI 的人设和行为规则 | 导演的剧本 |
| 用户 | `user` | 你发送的问题或指令 | 观众的提问 |
| 助手 | `assistant` | AI 的回复 | 演员的回答 |

```python
messages = [
    {"role": "system", "content": "你是一个专业的Python老师"},
    {"role": "user", "content": "什么是列表推导式？"},
    {"role": "assistant", "content": "列表推导式是Python中创建列表的简洁方式..."},
    {"role": "user", "content": "能给个例子吗？"}  # 多轮对话
]
```

### 2. Temperature 参数

Temperature 控制 AI 回答的"创造性"：

- **Temperature = 0**：AI 变成"老实人"，回答最确定、最保守的答案
- **Temperature = 1**：AI 变成"创意大师"，回答更随机、更有创造性
- **Temperature = 0.7**：平衡模式（推荐默认值）

类比：就像做菜放辣椒，0 是不辣，1 是特辣，0.7 是微辣。

### 3. Token 概念

Token 是 AI 处理文本的最小单位：
- 英文：大约 1 个单词 = 1 个 token
- 中文：大约 1-2 个汉字 = 1 个 token
- API 调用按 token 数量计费

类比：Token 就像手机流量，用得越多，费用越高。

### 4. client.chat.completions.create() 参数

```python
from openai import OpenAI

client = OpenAI(api_key="your-api-key")

response = client.chat.completions.create(
    model="gpt-4o-mini",         # 推荐模型：gpt-4o-mini（性价比高）或 gpt-4o（更强）
    messages=[                    # 消息列表
        {"role": "user", "content": "你好"}
    ],
    temperature=0.7,             # 创造性参数
    max_tokens=1000,             # 最大输出 token 数
)
```

> **模型选择建议**：Chat 场景推荐 `gpt-4o-mini`（便宜、快、够用）或 `gpt-4o`（更强但更贵）。不建议再使用 gpt-3.5-turbo。

## 在 RAG 项目中的应用场景

1. **最终答案生成**：RAG 检索到相关文档后，用 Chat Completion API 将检索结果和用户问题一起发送给 LLM，生成最终答案
2. **查询改写**：用 LLM 对用户的原始问题进行改写，提高检索效果
3. **答案摘要**：对检索到的多个文档进行总结
4. **多轮对话**：在 RAG 对话系统中维护对话历史，实现上下文连贯的问答

## 面试常问问题

### Q1: system、user、assistant 三个角色有什么区别？为什么不直接用 user？

**答**：system 角色用于设定 AI 的"人设"和行为规范，它对整个对话都有指导作用。user 是用户输入，assistant 是 AI 回复。使用 system 角色的好处是可以把"指令"和"内容"分离，让 AI 始终遵循特定的行为模式，而不需要每次都在 user 消息里重复说明规则。

### Q2: Temperature 设为 0 和设为 1 有什么区别？在 RAG 中通常设为多少？

**答**：Temperature = 0 时，模型会选择概率最高的词，输出最确定、最保守；Temperature = 1 时，模型会更随机地选择词，输出更有创造性但可能不稳定。在 RAG 项目中，通常设为 0 或较低值（如 0.1-0.3），因为我们希望答案基于检索到的文档准确生成，而不是"自由发挥"。

### Q3: Token 限制对你有什么影响？你怎么处理超长文档？

**答**：Token 是大语言模型处理文本的最小单位，英文约 1 单词 = 1 token，中文约 1-2 汉字 = 1 token。Token 限制的影响：1）计费依据，API 按 token 数量收费；2）上下文窗口限制，每个模型有最大 token 限制（如 gpt-4o-mini 支持 128k token）；3）性能影响，token 越多，响应越慢。

处理超长文档的方法：1）**文本切分**——将长文档按段落或语义切成小块（chunk），每块控制在模型上下文窗口内；2）**检索增强**——不把整篇文档发给模型，而是先检索最相关的片段，只把相关片段作为上下文；3）**摘要压缩**——先用 LLM 对长文档生成摘要，再用摘要作为上下文；4）**滑动窗口**——对超长文本分段处理，每段保留前后重叠部分以保持上下文连贯。RAG 系统中最常用的是"文本切分 + 检索增强"组合。
