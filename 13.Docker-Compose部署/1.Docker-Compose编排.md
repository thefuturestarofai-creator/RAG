# Docker Compose 编排

## 一句话理解

Docker Compose 就像一个"导演"——它负责指挥多个"演员"（容器）各就各位、互相配合，一起演出一台"戏"（应用）。你只需要写一个"剧本"（docker-compose.yml），导演就会自动把一切安排好。

---

## 一、为什么需要 Docker Compose？

### 没有 Docker Compose 的痛苦

```bash
# 手动启动 3 个容器，还要配置网络、端口、环境变量...
docker network create rag-network
docker run -d --name chromadb --network rag-network -p 8001:8000 chromadb/chroma
docker run -d --name rag-app --network rag-network -p 8000:8000 \
  -e CHROMA_HOST=chromadb \
  -e OPENAI_API_KEY=sk-xxx \
  rag-app
# 每次都要敲这么多命令，容易出错
```

### 有了 Docker Compose

```bash
# 一条命令搞定
docker-compose up -d
```

**类比**：没有 Compose 像自己一个个叫外卖（每个容器单独启动），有 Compose 像点了一个套餐（一个文件描述所有服务，一键启动）。

---

## 二、docker-compose.yml 核心概念

### 1. services — 服务定义

每个 service 对应一个容器：

```yaml
services:
  # 服务名：rag-app
  rag-app:
    build: .                    # 从当前目录的 Dockerfile 构建
    ports:
      - "8000:8000"            # 端口映射：宿主机:容器
    environment:
      - OPENAI_API_KEY=sk-xxx  # 环境变量
    depends_on:
      - chromadb               # 依赖关系：先启动 chromadb

  chromadb:
    image: chromadb/chroma:latest  # 使用现成的镜像
    ports:
      - "8001:8000"
    volumes:
      - chroma_data:/chroma/chroma  # 数据持久化
```

### 2. volumes — 数据持久化

```yaml
volumes:
  chroma_data:    # 命名卷，数据保存在 Docker 管理的目录中
```

**为什么需要 volumes？**
- 容器删除后数据会丢失
- volumes 把数据存在宿主机上，容器重建后数据还在

**类比**：容器像酒店房间（退房就没了），volumes 像保险柜（东西锁在里面，换房间也不丢）。

### 3. networks — 网络

```yaml
networks:
  rag-network:
    driver: bridge
```

同一网络中的容器可以用服务名互相访问：
```python
# rag-app 容器中访问 chromadb 容器
import chromadb
client = chromadb.HttpClient(host="chromadb", port=8000)
# 直接用服务名 "chromadb" 作为主机名
```

### 4. depends_on — 依赖关系

```yaml
services:
  rag-app:
    depends_on:
      chromadb:
        condition: service_healthy  # 等 chromadb 健康后再启动
```

## 三、RAG 项目的 Docker Compose 配置

### 服务架构

```
┌──────────────┐     ┌──────────────┐
│   rag-app    │────>│   chromadb   │
│  (FastAPI)   │     │ (向量数据库)  │
│  端口: 8000  │     │  端口: 8001  │
└──────────────┘     └──────────────┘
```

### 完整的 docker-compose.yml

```yaml
version: "3.8"

services:
  # ---- RAG 应用服务 ----
  rag-app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: rag-app
    ports:
      - "8000:8000"
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - OPENAI_BASE_URL=${OPENAI_BASE_URL}
      - OPENAI_MODEL=${OPENAI_MODEL}
      - CHROMA_HOST=chromadb
      - CHROMA_PORT=8000
    volumes:
      - ./data:/app/data           # 挂载数据目录
      - ./logs:/app/logs           # 挂载日志目录
    depends_on:
      chromadb:
        condition: service_healthy
    restart: unless-stopped         # 自动重启策略
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  # ---- ChromaDB 向量数据库 ----
  chromadb:
    image: chromadb/chroma:latest
    container_name: chromadb
    ports:
      - "8001:8000"
    volumes:
      - chroma_data:/chroma/chroma
    environment:
      - IS_PERSISTENT=TRUE
      - ANONYMIZED_TELEMETRY=FALSE
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/v1/heartbeat"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  chroma_data:
    driver: local
```

## 四、环境变量管理

### 方式一：.env 文件

```bash
# .env 文件
OPENAI_API_KEY=sk-your-key
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-3.5-turbo
```

docker-compose.yml 中引用：
```yaml
environment:
  - OPENAI_API_KEY=${OPENAI_API_KEY}
```

### 方式二：env_file 指令

```yaml
services:
  rag-app:
    env_file:
      - .env
```

## 五、常用命令

```bash
# 启动所有服务（后台运行）
docker-compose up -d

# 查看运行状态
docker-compose ps

# 查看日志
docker-compose logs -f rag-app

# 停止所有服务
docker-compose down

# 停止并删除数据卷
docker-compose down -v

# 重新构建并启动
docker-compose up -d --build

# 进入容器
docker-compose exec rag-app bash
```

## 六、面试常问问题

### Q1: Docker Compose 是什么？和 Docker 有什么区别？
**答**：Docker 用于构建和运行单个容器，Docker Compose 用于编排多个容器。Docker Compose 通过一个 YAML 文件定义所有服务、网络和卷，一条命令就能启动整个应用栈。

### Q2: docker-compose.yml 中的 depends_on 有什么作用？
**答**：定义服务之间的启动依赖关系。比如 rag-app 依赖 chromadb，Compose 会先启动 chromadb 再启动 rag-app。配合 condition: service_health 还能等待依赖服务健康后再启动。

### Q3: 为什么需要 volumes？
**答**：容器是临时的，删除后数据会丢失。volumes 把数据持久化到宿主机，即使容器重建数据也不会丢。比如向量数据库的数据、日志文件等都需要 volumes。

### Q4: Docker 容器之间怎么通信？
**答**：Compose 会自动创建一个网络，同一网络中的容器可以用服务名作为主机名互相访问。比如 rag-app 中用 "chromadb" 作为主机名就能访问 ChromaDB 服务。

### Q5: healthcheck 有什么作用？
**答**：定期检查容器是否健康。如果健康检查失败，容器会被标记为不健康，配合 depends_on 的 condition: service_healthy 可以确保依赖服务真正可用后再启动下游服务。
