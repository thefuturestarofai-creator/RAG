# 生产部署考虑

## 一句话理解

开发环境像"排练室"——怎么方便怎么来；生产环境像"正式演出"——灯光、音响、安全、备份都要到位。这一章就是告诉你从"排练室"走上"舞台"需要准备什么。

## 一、健康检查（Health Check）

### 为什么需要健康检查？

```
没有健康检查：
  容器挂了 → 你不知道 → 用户报错 → 你才发现

有健康检查：
  容器挂了 → 自动检测 → 自动重启 → 用户无感知
```

### Dockerfile 中的健康检查

```dockerfile
# 健康检查：每30秒检查一次 HTTP 服务
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1
```

### docker-compose.yml 中的健康检查

```yaml
services:
  rag-app:
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s      # 检查间隔
      timeout: 10s        # 超时时间
      retries: 3          # 失败重试次数
      start_period: 40s   # 启动等待时间
```

### FastAPI 健康检查接口

```python
@app.get("/health")
async def health():
    """健康检查接口"""
    return {
        "status": "healthy",
        "timestamp": datetime.now().isoformat(),
        "version": "1.0.0"
    }
```

## 二、日志管理

### 容器日志策略

```yaml
services:
  rag-app:
    logging:
      driver: "json-file"    # 日志驱动
      options:
        max-size: "10m"      # 单个日志文件最大 10MB
        max-file: "3"        # 最多保留 3 个日志文件
```

### 日志级别配置

```python
import logging

# 根据环境变量设置日志级别
log_level = os.getenv("LOG_LEVEL", "INFO")
logging.basicConfig(level=getattr(logging, log_level))
```

### 结构化日志

```python
import json
from datetime import datetime

def structured_log(level, message, **kwargs):
    """输出 JSON 格式的结构化日志"""
    log_entry = {
        "timestamp": datetime.now().isoformat(),
        "level": level,
        "message": message,
        **kwargs
    }
    print(json.dumps(log_entry, ensure_ascii=False))

# 使用
structured_log("INFO", "用户查询", question="什么是RAG?", user_id="12345")
```

## 三、资源限制

### 内存限制

```yaml
services:
  rag-app:
    deploy:
      resources:
        limits:
          memory: 2G        # 最大内存
          cpus: "2.0"       # 最大 CPU 核数
        reservations:
          memory: 512M      # 预留内存
          cpus: "0.5"       # 预留 CPU
```

### 为什么需要资源限制？

- 防止单个容器占用所有资源
- 确保其他服务正常运行
- 避免内存泄漏导致系统崩溃

## 四、安全考虑

### 1. 不要以 root 运行

```dockerfile
# 创建非 root 用户
RUN useradd -m -u 1000 appuser
USER appuser
```

### 2. 环境变量安全

```yaml
# ❌ 不推荐：直接写在 docker-compose.yml 中
environment:
  - OPENAI_API_KEY=sk-xxx

# ✅ 推荐：使用 .env 文件或 Docker secrets
env_file:
  - .env
```

### 3. 最小化镜像

```dockerfile
# 使用 slim 镜像
FROM python:3.11-slim

# 只安装必要的依赖
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
```

## 五、部署脚本

### 完整的部署流程

```bash
#!/bin/bash
# deploy.sh - RAG 系统部署脚本

set -e  # 遇到错误立即退出

echo "=== RAG 系统部署 ==="

# 1. 检查环境
echo "[1/5] 检查环境..."
if [ ! -f .env ]; then
    echo "错误: .env 文件不存在"
    exit 1
fi

# 2. 拉取最新代码
echo "[2/5] 拉取最新代码..."
git pull origin main

# 3. 构建镜像
echo "[3/5] 构建 Docker 镜像..."
docker-compose build

# 4. 停止旧容器
echo "[4/5] 停止旧容器..."
docker-compose down

# 5. 启动新容器
echo "[5/5] 启动新容器..."
docker-compose up -d

# 6. 检查状态
echo "等待服务启动..."
sleep 10
docker-compose ps

echo "=== 部署完成 ==="
```

## 六、面试常问问题

### Q1: 生产环境部署需要考虑哪些方面？
**答**：主要包括：1）健康检查——自动检测服务状态；2）日志管理——集中收集和分析日志；3）资源限制——防止资源耗尽；4）安全——最小权限、密钥管理；5）备份——数据持久化和定期备份；6）监控——实时监控服务状态。

### Q2: Docker 的健康检查怎么做？
**答**：在 docker-compose.yml 中配置 healthcheck，指定检查命令（如 curl 健康接口）、间隔时间、超时时间和重试次数。配合 depends_on 的 condition: service_healthy，可以确保依赖服务健康后再启动下游服务。

### Q3: 如何管理 Docker 容器的日志？
**答**：通过 logging 配置日志驱动和选项。推荐使用 json-file 驱动，设置 max-size（单文件大小限制）和 max-file（文件数量限制）防止日志撑满磁盘。生产环境可以用 ELK（Elasticsearch + Logstash + Kibana）集中管理日志。

### Q4: 为什么要限制容器的资源？
**答**：防止单个容器占用所有 CPU 和内存，导致其他服务无法运行。通过 deploy.resources.limits 设置上限，reservations 设置预留。对于 RAG 应用，Embedding 和 LLM 调用可能消耗较多内存，需要合理设置。

### Q5: Docker 容器的安全最佳实践有哪些？
**答**：1）不以 root 用户运行；2）使用最小基础镜像（如 python:3.11-slim）；3）敏感信息用 secrets 或 .env 文件，不写在代码里；4）定期更新基础镜像修复漏洞；5）只暴露必要的端口。
