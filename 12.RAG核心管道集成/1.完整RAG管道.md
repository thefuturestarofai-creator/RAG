# 完整 RAG 管道

## 一句话理解

RAG 管道就像一条"流水线"——原材料（文档）从一端进去，经过加工（加载→切分→向量化→存储），然后当用户提问时，流水线反向工作（检索→组装Prompt→LLM生成→返回回答）。这一章就是把前面学的所有零件组装成一条完整的流水线。

---

## 一、完整 RAG 管道流程图

```
=== 离线阶段：文档入库 ===

  原始文档（PDF/TXT/MD）
       │
       ▼
  ┌─────────┐
  │ 文档加载 │  TextLoader / PDFPlumberLoader
  └────┬────┘
       │ List[Document]
       ▼
  ┌─────────┐
  │ 文档清洗 │  去除特殊字符、多余空白
  └────┬────┘
       │ List[Document]
       ▼
  ┌─────────┐
  │ 文档切分 │  RecursiveCharacterTextSplitter
  └────┬────┘
       │ List[Document] (chunks)
       ▼
  ┌─────────┐
  │ Embedding│  text-embedding-ada-002
  └────┬────┘
       │ List[float] (向量)
       ▼
  ┌─────────┐
  │ 向量存储 │  Chroma / Milvus
  └─────────┘


=== 在线阶段：用户提问 ===

  用户问题："什么是RAG？"
       │
       ▼
  ┌─────────┐
  │ 问题向量化│  Embedding
  └────┬────┘
       │ [0.1, 0.3, ...]
       ▼
  ┌─────────┐
  │ 相似度检索│  从向量数据库找 top-k 相关片段
  └────┬────┘
       │ List[Document]
       ▼
  ┌─────────┐
  │ Prompt   │  模板 + 问题 + 参考资料
  │ 组装     │
  └────┬────┘
       │ str (完整 Prompt)
       ▼
  ┌─────────┐
  │ LLM 生成 │  GPT-3.5-turbo / GPT-4
  └────┬────┘
       │ str (回答)
       ▼
  ┌─────────┐
  │ 后处理   │  格式化、来源标注、安全检查
  └────┬────┘
       │
       ▼
  返回给用户
```

## 二、各环节详解

### 1. 文档加载

```python
# 根据文件类型选择加载器
loader = TextLoader("data.txt", encoding="utf-8")
documents = loader.load()
# 结果: [Document(page_content="...", metadata={"source": "data.txt"})]
```

### 2. 文档清洗

```python
# 去除特殊字符、多余空白
cleaned_docs = preprocessor.process(documents)
```

### 3. 文档切分

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,      # 每个 chunk 最大 500 字符
    chunk_overlap=50,    # 相邻 chunk 重叠 50 字符
    separators=["\n\n", "\n", "。", "！", "？", " ", ""]
)
chunks = splitter.split_documents(cleaned_docs)
```

### 4. Embedding 向量化

```python
embeddings = OpenAIEmbeddings(
    model="text-embedding-ada-002",
    openai_api_key=API_KEY,
    openai_api_base=BASE_URL,
)
# 将文本转为向量
vector = embeddings.embed_query("什么是RAG？")
# vector = [0.1, -0.3, 0.5, ...] (1536维)
```

### 5. 向量存储

```python
# 创建向量数据库并存储
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./data/chroma_db",
    collection_name="rag_collection",
)
```

### 6. 相似度检索

```python
# 根据用户问题检索相关文档
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 3}
)
relevant_docs = retriever.invoke("什么是RAG？")
```

### 7. Prompt 组装

```python
template = """请根据以下参考资料回答问题。
如果资料中没有相关信息，请如实说明。

参考资料：
{context}

问题：{question}
回答："""

prompt = ChatPromptTemplate.from_template(template)
messages = prompt.format_messages(
    context="\n\n---\n\n".join([doc.page_content for doc in relevant_docs]),
    question="什么是RAG？"
)
```

### 8. LLM 生成

```python
llm = ChatOpenAI(
    model="gpt-3.5-turbo",
    openai_api_key=API_KEY,
    openai_api_base=BASE_URL,
    temperature=0.3,
)
response = llm.invoke(messages)
answer = response.content
```

## 三、LangChain 的链式调用

LangChain 提供了 LCEL（LangChain Expression Language）来串联各个组件：

```python
from langchain.schema.runnable import RunnablePassthrough
from langchain.schema.output_parser import StrOutputParser

# 构建 RAG 链
rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# 使用
answer = rag_chain.invoke("什么是RAG？")
```

**类比**：LCEL 就像流水线上的管道（`|`），数据从左边流入，经过每个处理环节，从右边流出。

## 四、关键设计决策

### 1. chunk_size 怎么选？

| chunk_size | 优点 | 缺点 | 适用场景 |
|-----------|------|------|---------|
| 200-300 | 检索精确 | 可能丢失上下文 | FAQ、短文档 |
| 500-800 | 平衡 | - | 通用场景 |
| 1000+ | 上下文完整 | 检索可能不精确 | 长文档、技术文档 |

### 2. top_k 怎么选？

- 太小（1-2）：可能遗漏信息
- 太大（10+）：引入噪声，浪费 token
- 推荐：3-5 作为起点

### 3. temperature 怎么选？

- 0：最确定性，适合事实性问答
- 0.3：轻微随机，回答更自然
- 0.7+：较高随机性，适合创意写作

## 五、面试常问问题

### Q1: 完整的 RAG 管道包括哪些步骤？
**答**：分为离线和在线两个阶段。离线：文档加载→清洗→切分→Embedding→向量存储。在线：用户问题→问题向量化→相似度检索→Prompt组装→LLM生成→后处理返回。

### Q2: 什么是 LCEL？
**答**：LCEL（LangChain Expression Language）是 LangChain 的链式调用语法，用管道符 `|` 串联各个组件。比如 `chain = retriever | prompt | llm | output_parser`。优点是声明式、支持流式输出、自动并行。

### Q3: chunk_size 和 top_k 怎么调优？
**答**：chunk_size 从 500 开始，根据检索效果调整——如果回答经常缺少上下文就增大，如果检索不精确就减小。top_k 从 3 开始，如果遗漏信息就增大，如果引入噪声就减小。建议用一组测试问题来评估不同参数的效果。

### Q4: RAG 和微调（Fine-tuning）有什么区别？
**答**：RAG 通过检索外部知识来增强回答，适合知识频繁更新的场景，不需要重新训练模型。微调通过训练让模型"记住"知识，适合需要特定风格或格式的场景，但知识更新需要重新训练。两者可以结合使用。

### Q5: 如何评估 RAG 系统的效果？
**答**：三个维度：1）检索质量——检索到的文档是否相关（用 precision@k、recall@k 衡量）；2）生成质量——回答是否准确、完整（人工评估或 LLM 评估）；3）端到端效果——最终回答是否正确（用标准问答对测试）。
