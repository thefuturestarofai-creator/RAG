# 来源追溯与流式输出

## 一句话理解

来源追溯就像学术论文的"参考文献"——你说的每句话都要标注出处，让用户知道信息来自哪个文件、第几页，增加可信度。流式输出就像"打字机效果"——回答一个字一个字地出现，用户体验更好。

---

## 一、来源追溯

### 为什么需要来源追溯？

1. **可信度**：用户可以验证回答的来源
2. **可追溯**：出了问题可以定位到具体文档
3. **合规性**：某些场景要求标注信息来源

### 实现方式

#### 1. Document 的 metadata 设计

在文档入库时，就在 metadata 中记录来源信息：

```python
Document(
    page_content="RAG是检索增强生成...",
    metadata={
        "source": "data/技术文档/RAG介绍.pdf",  # 来源文件
        "page": 3,                               # 页码
        "chunk_index": 5,                        # 第几个 chunk
        "title": "RAG技术详解",                   # 文档标题
        "author": "张三",                         # 作者
        "date": "2024-01-15",                    # 日期
    }
)
```

#### 2. 检索时返回 metadata

```python
# 相似度搜索，返回文档和相关度分数
results = vectorstore.similarity_search_with_score(
    query="什么是RAG？",
    k=3
)

for doc, score in results:
    print(f"来源: {doc.metadata['source']}")
    print(f"页码: {doc.metadata.get('page', '未知')}")
    print(f"相关度: {score:.4f}")
    print(f"内容: {doc.page_content[:100]}...")
    print()
```

#### 3. 在 Prompt 中要求 LLM 标注来源

```python
template = """请根据以下参考资料回答问题，并标注信息来源。

要求：
1. 在回答中用 [来源X] 标注每条信息的出处
2. 在回答末尾列出完整的参考来源

参考资料：
{context}

问题：{question}
回答："""

# 构造包含来源信息的 context
context_parts = []
for i, doc in enumerate(relevant_docs, 1):
    source = doc.metadata.get("source", "未知")
    page = doc.metadata.get("page", "")
    page_info = f"第{page}页" if page else ""
    context_parts.append(
        f"[来源{i}] 来自 {source} {page_info}\n{doc.page_content}"
    )
context = "\n\n---\n\n".join(context_parts)
```

#### 4. 返回结构化结果

```python
from pydantic import BaseModel
from typing import List

class Source(BaseModel):
    """来源信息"""
    file_name: str      # 文件名
    page: int = None    # 页码
    relevance: float    # 相关度
    snippet: str        # 原文片段

class RAGResponse(BaseModel):
    """RAG 响应"""
    answer: str                # 回答内容
    sources: List[Source]      # 来源列表
    question: str              # 原始问题
```

## 二、流式输出

### 为什么需要流式输出？

**对比**：
- 非流式：用户等待 5 秒 → 一次性看到完整回答（体验差）
- 流式：用户等待 0.5 秒 → 逐字看到回答出现（体验好）

**类比**：非流式像等快递（等很久一次收到），流式像看直播（实时看到内容）。

### SSE（Server-Sent Events）原理

```
客户端（浏览器）              服务端（FastAPI）
    │                            │
    │  GET /stream-query         │
    │  Accept: text/event-stream │
    │ ─────────────────────────> │
    │                            │
    │  data: {"token": "RAG"}    │
    │ <───────────────────────── │
    │                            │
    │  data: {"token": "是"}     │
    │ <───────────────────────── │
    │                            │
    │  data: {"token": "一种"}   │
    │ <───────────────────────── │
    │                            │
    │  data: [DONE]              │
    │ <───────────────────────── │
```

### FastAPI 流式输出实现

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import json

app = FastAPI()

async def generate_stream(question: str):
    """生成流式响应"""
    # 调用 LLM 的流式接口
    for chunk in llm.stream(messages):
        token = chunk.content
        # SSE 格式：data: {json}\n\n
        yield f"data: {json.dumps({'token': token}, ensure_ascii=False)}\n\n"

    # 结束标记
    yield "data: [DONE]\n\n"

@app.get("/stream-query")
async def stream_query(question: str):
    """流式问答接口"""
    return StreamingResponse(
        generate_stream(question),
        media_type="text/event-stream",  # SSE 的 MIME 类型
        headers={
            "Cache-Control": "no-cache",      # 不缓存
            "Connection": "keep-alive",       # 保持连接
            "X-Accel-Buffering": "no",        # 禁用 Nginx 缓冲
        }
    )
```

### 前端接收流式数据

```javascript
// JavaScript 接收 SSE
const eventSource = new EventSource('/stream-query?question=什么是RAG');

eventSource.onmessage = function(event) {
    if (event.data === '[DONE]') {
        eventSource.close();
        return;
    }
    const data = JSON.parse(event.data);
    document.getElementById('answer').innerHTML += data.token;
};

eventSource.onerror = function() {
    eventSource.close();
};
```

## 三、带来源追溯的流式输出

将两者结合，先返回来源信息，再流式返回回答：

```python
async def generate_with_sources(question: str):
    # 1. 先检索相关文档
    docs = retriever.invoke(question)

    # 2. 先返回来源信息
    sources = [
        {
            "file": doc.metadata.get("source", ""),
            "page": doc.metadata.get("page", ""),
            "snippet": doc.page_content[:100],
        }
        for doc in docs
    ]
    yield f"data: {json.dumps({'sources': sources}, ensure_ascii=False)}\n\n"

    # 3. 再流式返回回答
    for chunk in llm.stream(messages):
        yield f"data: {json.dumps({'token': chunk.content}, ensure_ascii=False)}\n\n"

    yield "data: [DONE]\n\n"
```

## 四、面试常问问题

### Q1: 如何在 RAG 回答中标注来源？
**答**：三步：1）在文档入库时，在 metadata 中记录来源信息（文件名、页码等）；2）检索时返回 metadata；3）在 Prompt 中要求 LLM 标注来源，并将 metadata 信息传给 LLM。

### Q2: 什么是 SSE？和 WebSocket 有什么区别？
**答**：SSE（Server-Sent Events）是服务端向客户端单向推送数据的协议，基于 HTTP。WebSocket 是双向通信协议。RAG 场景用 SSE 就够了，因为只需要服务端向客户端推送 token。

### Q3: 流式输出的实现原理是什么？
**答**：服务端调用 LLM 的流式接口（stream），每收到一个 token 就通过 SSE 推送给客户端。客户端用 EventSource API 接收并逐字显示。FastAPI 用 StreamingResponse 实现。

### Q4: 如何设计 Document 的 metadata？
**答**：至少包含 source（来源文件）和 page（页码）。可以扩展：title（标题）、author（作者）、date（日期）、chunk_index（切分序号）。metadata 应该支持过滤查询（如只在某个文件范围内搜索）。

### Q5: 流式输出中如何处理来源信息？
**答**：两种方式：1）先发送一个包含来源列表的特殊消息，再流式发送回答 token；2）在回答的最后一条消息中附带来源。推荐方式 1，因为前端可以先展示来源卡片，再逐字展示回答。
