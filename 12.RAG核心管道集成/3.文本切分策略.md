# 文本切分策略（Chunking Strategies）

> 学习目标：理解 RAG 系统中文本切分的核心概念，掌握 4 种主流切分策略的原理、优缺点和适用场景。
> 文本切分是 RAG 管道中最关键的环节之一，直接影响检索质量和回答准确性。

---

## 0. 为什么需要文本切分？

**通俗理解**：想象你在图书馆找一本书的某个知识点。

- **不切分**：把整本书塞给你 → 信息太多，找不到重点
- **切分太细**：每次只给你一句话 → 缺乏上下文，理解困难
- **切分合适**：给你相关的几个段落 → 信息完整，重点突出

**技术原因**：
1. **Embedding 模型有 Token 限制**：大多数模型最多处理 512 或 8192 个 Token
2. **检索精度**：小块文本更容易匹配查询意图
3. **成本控制**：发送给 LLM 的 Token 越少，成本越低
4. **上下文窗口**：LLM 的上下文窗口有限，不能把所有文档都塞进去

---

## 1. 四种主流切分策略

### 1.1 固定长度切分（Fixed-size Chunking）

**原理**：按照固定的字符数或 Token 数切分，不考虑文本内容。

**类比**：就像用尺子量布，每隔 50 厘米剪一刀，不管图案在哪里。

```python
# 伪代码示例
text = "很长的文档内容..."
chunk_size = 500  # 每个 chunk 500 个字符
chunks = [text[i:i+chunk_size] for i in range(0, len(text), chunk_size)]
```

**优点**：
- 实现简单，速度最快
- Chunk 大小一致，便于管理
- 适合结构化程度低的文本

**缺点**：
- 可能在句子中间断开，破坏语义完整性
- 不考虑文本的自然边界（段落、章节）
- 检索时可能返回不完整的答案

**适用场景**：
- 快速原型开发
- 文本结构不明确的情况
- 对切分质量要求不高的场景

---

### 1.2 递归字符切分（Recursive Character Splitting）

**原理**：按照分隔符的优先级递归切分。先尝试用段落分隔符（`\n\n`），如果还是太长，就用换行符（`\n`），再不行就用句号（`.`），最后用空格。

**类比**：就像切蛋糕，先尝试沿着纹路切（段落），如果一块太大，再切小一点（句子），直到大小合适。

```python
# LangChain 的实现方式
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]  # 分隔符优先级
)
chunks = splitter.split_text(document)
```

**分隔符优先级**（从高到低）：
1. `\n\n` —— 段落分隔
2. `\n` —— 换行
3. `。！？` —— 句号、感叹号、问号
4. `，,` —— 逗号
5. ` ` —— 空格
6. `""` —— 最后才按字符切

**优点**：
- 尊重文本的自然结构
- 保持语义完整性
- 是 LangChain 的默认推荐方法
- 适用性广，大多数场景表现良好

**缺点**：
- 需要调参（chunk_size、chunk_overlap、separators）
- 对于特殊格式（如代码、表格）可能效果不佳

**适用场景**：
- 通用文档处理（推荐首选）
- 文章、报告、论文等结构化文本
- RAG 系统的标准选择

---

### 1.3 语义切分（Semantic Chunking）

**原理**：使用 Embedding 模型计算相邻句子的语义相似度，在语义发生明显变化的地方切分。

**类比**：就像听音乐时，你能感觉到什么时候换了一首歌——旋律、节奏突然变了。

```python
# 语义切分的核心逻辑
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

splitter = SemanticChunker(
    OpenAIEmbeddings(),
    breakpoint_threshold_type="percentile",  # 使用百分位数作为阈值
    breakpoint_threshold_amount=75           # 75% 分位数
)
chunks = splitter.split_text(document)
```

**切分逻辑**：
1. 先把文本分成句子
2. 计算相邻句子的 Embedding 相似度
3. 当相似度低于阈值时，在此处切分

**优点**：
- 切分点符合语义边界
- 每个 chunk 主题相对单一
- 检索时匹配精度最高

**缺点**：
- 需要调用 Embedding 模型，成本高
- 处理速度慢
- Chunk 大小不一致
- 依赖 Embedding 模型的质量

**适用场景**：
- 对检索精度要求极高的场景
- 主题频繁切换的文档
- 有预算支持 Embedding 调用的项目

---

### 1.4 按文档结构切分（Document Structure Splitting）

**原理**：根据文档的固有结构（标题、章节、段落）进行切分。

**类比**：就像按照书的目录来分章节，每一章是一个自然的知识单元。

```python
# 按 Markdown 标题切分
from langchain.text_splitter import MarkdownHeaderTextSplitter

headers_to_split_on = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]

splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
chunks = splitter.split_text(markdown_document)

# 按 HTML 标签切分
from langchain.text_splitter import HTMLHeaderTextSplitter

headers_to_split_on = [("h1", "Header 1"), ("h2", "Header 2")]
splitter = HTMLHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
```

**优点**：
- 保留文档的层级结构
- 自然的语义边界
- 可以保留元数据（如章节标题）
- 适合有明确结构的文档

**缺点**：
- 依赖文档的结构质量
- 不同格式需要不同的 Splitter
- 结构不规范的文档效果差
- Chunk 大小差异可能很大

**适用场景**：
- Markdown 文档
- HTML 页面
- 有明确标题层级的报告、论文
- 技术文档、API 文档

## 2. 四种策略对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 | 推荐指数 |
|------|------|------|------|----------|----------|
| **固定长度** | 按字符数切分 | 简单快速 | 破坏语义 | 原型开发 | ★★☆☆☆ |
| **递归字符** | 按分隔符优先级 | 通用性好 | 需要调参 | 通用场景 | ★★★★★ |
| **语义切分** | 按语义相似度 | 精度最高 | 成本高、慢 | 高精度需求 | ★★★★☆ |
| **文档结构** | 按标题/章节 | 保留结构 | 依赖格式 | 结构化文档 | ★★★★☆ |

## 3. Chunk Size 和 Chunk Overlap 调优指南

### 3.1 Chunk Size（块大小）

**什么是 Chunk Size？**
每个文本块的最大长度（字符数或 Token 数）。

**调优原则**：

| Chunk Size | 效果 | 适用场景 |
|------------|------|----------|
| **100-300** | 精度高，但上下文少 | 精确匹配（如 FAQ、代码片段） |
| **500-800** | 平衡精度和上下文 | 通用场景（推荐起点） |
| **1000-1500** | 上下文丰富，精度略降 | 需要理解长段落的场景 |
| **2000+** | 上下文最多，精度最低 | 摘要、总结类任务 |

**实际建议**：
- **起步值**：500 字符（中文）或 256 Token（英文）
- **测试方法**：用真实查询测试不同大小，看哪个返回的答案最准确
- **经验值**：大多数 RAG 系统使用 500-1000 字符效果最好

### 3.2 Chunk Overlap（重叠区域）

**什么是 Chunk Overlap？**
相邻两个文本块之间重叠的字符数。

**为什么需要重叠？**
- 防止重要信息被切到两个块的边界
- 保持上下文连贯性

**调优原则**：

| Overlap | 效果 | 适用场景 |
|---------|------|----------|
| **0** | 无重叠，可能丢失边界信息 | 快速处理，不关心边界 |
| **10-20%** | 轻微重叠，平衡成本 | 大多数场景（推荐） |
| **30-50%** | 大量重叠，信息冗余 | 边界信息很重要的场景 |

**实际建议**：
- **起步值**：Chunk Size 的 10-20%
- **例如**：Chunk Size = 500，Overlap = 50-100
- **不要太大**：Overlap 过大会导致存储和检索成本翻倍

### 3.3 调优流程

```
1. 确定 Chunk Size 起点（建议 500）
   ↓
2. 设置 Overlap 为 Chunk Size 的 15%
   ↓
3. 用真实查询测试
   ↓
4. 如果答案不完整 → 增大 Chunk Size
5. 如果答案太模糊 → 减小 Chunk Size
6. 如果边界信息丢失 → 增大 Overlap
   ↓
7. 重复 3-6，直到效果满意
```

## 4. Metadata 增强技巧

### 4.1 什么是 Metadata？

Metadata 是附加在每个 Chunk 上的额外信息，帮助更好地理解和检索文本。

**通俗理解**：就像图书馆的书签，告诉你这段内容属于哪本书、哪个章节。

### 4.2 常用的 Metadata 类型

```python
metadata = {
    "source": "rag_guide.md",           # 来源文件
    "chapter": "第3章 文本切分",         # 所属章节
    "page": 15,                          # 页码
    "title": "Chunk Size 调优指南",      # 章节标题
    "author": "张三",                    # 作者
    "date": "2024-06-01",               # 日期
    "doc_type": "tutorial",              # 文档类型
    "language": "zh",                    # 语言
}
```

### 4.3 如何注入 Metadata？

**方法1：使用文档结构 Splitter**

```python
from langchain.text_splitter import MarkdownHeaderTextSplitter

# MarkdownHeaderTextSplitter 会自动把标题注入 metadata
headers_to_split_on = [
    ("#", "Header 1"),
    ("##", "Header 2"),
    ("###", "Header 3"),
]

splitter = MarkdownHeaderTextSplitter(headers_to_split_on=headers_to_split_on)
chunks = splitter.split_text(markdown_text)

# 每个 chunk 的 metadata 会包含它所属的标题
# 例如：{"Header 1": "第3章", "Header 2": "文本切分策略"}
```

**方法2：手动添加 Metadata**

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_text(document)

# 给每个 chunk 添加 metadata
for i, chunk in enumerate(chunks):
    chunk.metadata = {
        "source": "my_document.md",
        "chunk_index": i,
        "total_chunks": len(chunks),
    }
```

**方法3：使用文档 Loader 的 Metadata**

```python
from langchain_community.document_loaders import DirectoryLoader

loader = DirectoryLoader("./docs", glob="**/*.md")
docs = loader.load()

# 每个 doc 已经包含了 source 文件路径等 metadata
# 切分后这些 metadata 会自动传递给 chunks
```

### 4.4 Metadata 在检索中的应用

**过滤检索**：先用 Metadata 过滤，再做相似度搜索

```python
# 只搜索特定章节的内容
results = vectorstore.similarity_search(
    "什么是 Chunk Size？",
    filter={"Header 1": "第3章 文本切分"}
)

# 只搜索特定来源的文档
results = vectorstore.similarity_search(
    "如何部署 RAG？",
    filter={"source": "deployment_guide.md"}
)
```

## 5. 面试加分话术

### 5.1 展示系统性思维

> "在选择 Chunking 策略时，我会从三个维度考虑：
> 1. **文档特性**：文档是否有明确结构？内容主题是否频繁切换？
> 2. **查询特性**：用户的查询是精确的还是模糊的？需要多长的上下文？
> 3. **系统约束**：Embedding 调用的预算？对延迟的要求？
>
> 对于大多数场景，我会从递归字符切分开始，Chunk Size 500，Overlap 15%，然后根据实际效果调优。"

### 5.2 展示实践经验

> "在实际项目中，我发现单一策略往往不够。我的做法是：
> 1. 先用 MarkdownHeaderTextSplitter 按结构切分，保留章节标题作为 Metadata
> 2. 对于过长的 chunk，再用 RecursiveCharacterTextSplitter 二次切分
> 3. 给每个 chunk 注入来源、章节、页码等 Metadata，支持过滤检索
>
> 这种组合策略在我们的生产环境中，检索准确率提升了约 20%。"

### 5.3 展示对细节的理解

> "Chunk Overlap 的设置很关键。我做过对比实验：
> - 没有 Overlap 时，约 15% 的查询会丢失边界信息
> - 设置 15% 的 Overlap 后，这个问题基本解决
> - 但 Overlap 超过 30% 时，存储成本显著增加，但精度提升不明显
>
> 所以我一般用 15% 作为起点。"

## 6. 面试常问问题

### Q1：如何选择合适的 Chunk Size？

**参考答案**：
1. **没有万能值**，需要根据具体场景调优
2. **起步建议**：500 字符（中文）或 256 Token（英文）
3. **调优方法**：
   - 准备一组测试问题和标准答案
   - 用不同的 Chunk Size 运行 RAG
   - 对比哪个 Size 返回的答案最准确
4. **一般规律**：
   - 精确查询（如 FAQ）→ 小 Chunk（200-400）
   - 复杂查询（需要综合理解）→ 大 Chunk（800-1500）

### Q2：递归字符切分和语义切分有什么区别？什么时候用哪个？

**参考答案**：
- **递归字符切分**：按分隔符优先级切分，简单快速，是大多数场景的首选
- **语义切分**：按语义相似度切分，精度更高，但需要调用 Embedding 模型
- **选择建议**：
  - 预算有限、对延迟敏感 → 递归字符切分
  - 追求最高检索精度、有 Embedding 预算 → 语义切分
  - 实际项目中，递归字符切分 + Metadata 增强通常就够用了

### Q3：什么是 Chunk Overlap？为什么需要它？

**参考答案**：
- **定义**：相邻两个文本块之间重叠的字符数
- **为什么需要**：
  - 防止重要信息被切到边界，导致检索时丢失
  - 保持上下文连贯性
- **设置建议**：Chunk Size 的 10-20%
- **过大问题**：存储和检索成本翻倍，但精度提升有限

## 7. 总结

| 维度 | 推荐做法 |
|------|----------|
| **首选策略** | 递归字符切分（通用性最好） |
| **Chunk Size** | 500-800 字符起步，根据效果调优 |
| **Chunk Overlap** | Chunk Size 的 15% |
| **Metadata** | 注入来源、章节标题、页码 |
| **高级场景** | 语义切分（高精度需求）或结构切分（Markdown/HTML） |
| **调优方法** | 用真实查询测试，对比不同参数的检索效果 |
