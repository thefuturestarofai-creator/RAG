## 一、为什么 RAG 需要对话记忆？

### 1.1 一个真实的场景

想象你在和一个 RAG 系统对话：

```
用户：帮我介绍一下 LangChain 框架
系统：LangChain 是一个用于构建 LLM 应用的框架，主要包括以下核心组件：
      1. Model I/O：负责与语言模型交互
      2. Retrieval：负责文档检索
      3. Memory：负责对话记忆管理
      4. Chains：负责组件编排

用户：你刚才说的第三点是什么意思？   ← 追问！
系统：？？？
```

**问题**：如果没有对话记忆，系统不知道"第三点"是什么，因为它把每次提问都当作独立事件。

### 1.2 核心问题

RAG 系统默认是**无状态的**——每次提问都是一次全新的请求。但真实对话是**有上下文的**：

- 用户会追问："你刚才说的那个方法怎么用？"
- 用户会指代："它和另一个框架相比怎么样？"（"它"指代上文的 LangChain）
- 用户会总结："把上面的内容总结成表格"

**没有对话记忆 = 每次对话都失忆**，用户体验极差。

## 二、三种记忆模式对比

### 2.1 类比理解：笔记本

| 模式 | 类比 | 优点 | 缺点 |
|------|------|------|------|
| ConversationBufferMemory | **记所有笔记** —— 每页都保留 | 信息完整，不丢失 | 笔记本越来越厚，成本高 |
| ConversationSummaryMemory | **每页写总结** —— 定期压缩 | 长度可控，保留关键信息 | 细节可能丢失 |
| ConversationBufferWindowMemory | **只保留最近N页** —— 滑动窗口 | 简单高效，长度固定 | 早期对话完全遗忘 |

### 2.2 ConversationBufferMemory（全量保存）

```
对话历史：
[用户] 你好，介绍一下 RAG
[AI] RAG 是检索增强生成...
[用户] 它的核心流程是什么？
[AI] 核心流程包括：加载、切分、嵌入、存储、检索、生成
[用户] 能详细说说检索吗？
[AI] 检索阶段主要使用向量相似度...

→ 全部保存，一字不漏
```

**特点**：
- 把所有对话记录原封不动地保存
- Token 消耗随对话轮次线性增长
- 适合短对话场景（<10轮）

### 2.3 ConversationSummaryMemory（摘要压缩）

```
对话摘要：
"用户在了解 RAG 技术。已经讨论了 RAG 的定义（检索增强生成）、
核心流程（加载→切分→嵌入→存储→检索→生成），
最近在讨论检索阶段的向量相似度匹配。"

→ 压缩成一段摘要，保留关键信息
```

**特点**：
- 用 LLM 将对话历史压缩成摘要
- Token 消耗增长缓慢（摘要长度有限）
- 适合长对话场景（>20轮）
- 需要额外的 LLM 调用来生成摘要

### 2.4 ConversationBufferWindowMemory（滑动窗口）

```
窗口大小 k=2，只保留最近2轮：

[用户] 能详细说说检索吗？           ← 保留
[AI] 检索阶段主要使用向量相似度...   ← 保留
[用户] 它和 SQL 查询有什么区别？     ← 保留
[AI] 向量检索是语义匹配...           ← 保留

之前的对话全部丢弃。

→ 只看最近 N 轮
```

**特点**：
- 只保留最近 k 轮对话
- Token 消耗固定（取决于窗口大小 k）
- 适合对话主题经常切换的场景
- 早期信息完全丢失

### 2.5 三种模式对比总结

```
              Token消耗    信息保留    实现复杂度    适用场景
全量保存        线性增长     完整        简单        短对话
摘要压缩        缓慢增长     关键信息    较复杂      长对话
滑动窗口        固定         最近N轮     简单        主题切换频繁
```

## 三、如何在 RAG 管道中集成对话记忆

### 3.1 核心思路

将对话历史**注入到 Prompt 中**，让 LLM 在回答时能参考之前的对话内容。

```
传统 RAG Prompt：
"""
你是一个知识助手。根据以下参考资料回答用户问题。
参考资料：{context}
用户问题：{question}
"""

加入对话记忆后的 Prompt：
"""
你是一个知识助手。根据以下参考资料和对话历史回答用户问题。
对话历史：{chat_history}
参考资料：{context}
用户问题：{question}
"""
```

### 3.2 集成步骤

```
步骤1：用户提问
    ↓
步骤2：加载对话历史
    ↓
步骤3：用对话历史 + 用户问题进行检索（可选：改写查询）
    ↓
步骤4：检索相关文档
    ↓
步骤5：构造 Prompt = 系统指令 + 对话历史 + 检索结果 + 用户问题
    ↓
步骤6：LLM 生成回答
    ↓
步骤7：将 (用户问题, AI回答) 存入记忆
```

### 3.3 LangChain 代码实现

**ConversationBufferMemory 基本用法**：

```python
from langchain.memory import ConversationBufferMemory

# 创建记忆实例
memory = ConversationBufferMemory(
    return_messages=True,      # 返回消息对象而非字符串
    memory_key="chat_history", # 在 Prompt 中的变量名
)

# 手动添加对话
memory.save_context(
    {"input": "介绍一下 LangChain"},
    {"output": "LangChain 是一个用于构建 LLM 应用的框架..."}
)
memory.save_context(
    {"input": "它怎么安装？"},
    {"output": "可以通过 pip install langchain 安装..."}
)

# 读取对话历史
print(memory.load_memory_variables({}))
# 输出: {"chat_history": [HumanMessage(...), AIMessage(...), ...]}
```

**在 RAG 管道中集成对话记忆**：

```python
from langchain.memory import ConversationBufferMemory
from langchain.chains import ConversationalRetrievalChain
from langchain_openai import ChatOpenAI
from langchain_community.vectorstores import Chroma

# 1. 创建记忆
memory = ConversationBufferMemory(
    memory_key="chat_history",
    return_messages=True,
)

# 2. 创建 RAG 链（带对话记忆）
qa_chain = ConversationalRetrievalChain.from_llm(
    llm=ChatOpenAI(model="gpt-3.5-turbo", temperature=0.3),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    memory=memory,                    # 注入记忆
    return_source_documents=True,     # 返回来源文档
)

# 3. 多轮对话
result1 = qa_chain.invoke({"question": "RAG 是什么？"})
print(result1["answer"])

result2 = qa_chain.invoke({"question": "它的核心流程是什么？"})
# 系统会自动结合上一轮的对话历史来理解"它"指代 RAG
print(result2["answer"])

# 4. 查看对话历史
print(memory.load_memory_variables({}))
```

**手动实现带记忆的 RAG Prompt**：

```python
from langchain.memory import ConversationBufferMemory
from langchain.prompts import ChatPromptTemplate

memory = ConversationBufferMemory(memory_key="chat_history", return_messages=True)

# 包含对话历史的 Prompt 模板
template = """你是一个知识助手。根据以下参考资料和对话历史回答用户问题。

对话历史：
{chat_history}

参考资料：
{context}

用户问题：{question}
请回答："""

prompt = ChatPromptTemplate.from_template(template)

def rag_with_memory(question: str, retriever, llm, memory):
    """带对话记忆的 RAG 调用"""
    # 1. 检索
    docs = retriever.invoke(question)
    context = "\n\n---\n\n".join([d.page_content for d in docs])

    # 2. 加载对话历史
    history = memory.load_memory_variables({})

    # 3. 构造 Prompt
    messages = prompt.format_messages(
        chat_history=history["chat_history"],
        context=context,
        question=question,
    )

    # 4. 生成
    response = llm.invoke(messages)
    answer = response.content

    # 5. 保存到记忆
    memory.save_context({"input": question}, {"output": answer})

    return answer
```

### 3.4 进阶：对话历史改写查询

有时候用户的追问是不完整的，需要结合历史来理解：

```
用户之前问："介绍一下 LangChain"
用户现在问："它怎么安装？"

如果直接用"它怎么安装？"去检索，效果很差。
结合历史改写为："LangChain 怎么安装？" → 检索效果大幅提升！
```

这种技术叫**查询改写（Query Rewriting）**，和对话记忆配合使用效果更好。

## 五、面试常问问题

### Q1: RAG 系统如何实现多轮对话？

**参考答案**：
> RAG 系统通过 Memory 模块实现多轮对话。核心思路是将对话历史注入到 Prompt 中。常见实现有三种：全量保存（ConversationBufferMemory）适合短对话；摘要压缩（ConversationSummaryMemory）用 LLM 将历史压缩为摘要，适合长对话；滑动窗口（ConversationBufferWindowMemory）只保留最近 k 轮，适合主题切换频繁的场景。实际项目中，通常还会结合查询改写，将指代性问题改写为完整问题，提升检索效果。

### Q2: 对话历史太长导致 Token 超限怎么办？

**参考答案**：
> 有三种策略：第一，使用滑动窗口，只保留最近 k 轮对话；第二，使用摘要压缩，用 LLM 将历史对话压缩成摘要；第三，混合策略，最近 k 轮保留原文，更早的对话用摘要。实际项目中推荐第三种，既保证了近期对话的细节，又控制了 Token 消耗。另外，还可以对检索结果进行截断或压缩，进一步减少 Prompt 长度。

### Q3: 如何处理对话中的指代消解问题？

**参考答案**：
> 指代消解是对话系统的核心挑战。常见做法是查询改写（Query Rewriting）：利用 LLM 结合对话历史，将包含指代的问题改写为完整问题。例如，用户问"它怎么安装？"，结合历史中的"LangChain"，改写为"LangChain 怎么安装？"。这个改写后的查询再去检索，效果会好很多。实现上，可以在检索前加一个查询改写的步骤，Prompt 中要求 LLM 结合对话历史生成完整的查询。
