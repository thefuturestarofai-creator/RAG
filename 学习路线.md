### 1.2 HTTP 与 Web 基础概念

| 知识点 | 掌握程度 | 说明 |

| GET vs POST 方法 | **能区分 + 知道何时用** | GET 拿数据，POST 提交数据 |
| JSON 数据格式 | **熟练读写** | API 的"通用语言"，90% 的数据交换都靠它 |
| 状态码（200/404/500） | **知道含义** | 200 成功、404 找不到、500 服务器炸了 |
| 什么是 API / RESTful API | **理解概念** | "去餐厅点菜"类比：API 就是菜单，REST 就是点菜的规矩 |
| 什么是 localhost / 端口号 | **理解概念** | `127.0.0.1:8000` = 你家电脑的第 8000 号门 |

**⏱ 建议时长：3-5 天**

**💡 类比帮助理解：**
> 把 Web 服务想象成一家餐厅：**客户端**是顾客，**服务器**是厨房，**API** 是菜单，**HTTP 方法**是点菜方式（GET=看菜单，POST=下单），**JSON** 是菜品描述的格式，**状态码**是服务员的回复（200=菜好了，404=没这道菜，500=厨房着火了）。

### 2.1 OpenAI API — 大模型的"遥控器"

#### 核心学习知识点（20% 骨干）

| 概念 | 说明 |
|---|---|
| **Chat Completion API** | `client.chat.completions.create()` — 和大模型对话的唯一入口 |
| **Embedding API** | `client.embeddings.create()` — 把文本变成向量（一串数字） |
| **模型选择** | Chat 用 `gpt-4o-mini` / `gpt-4o`，Embedding 用 `text-embedding-3-small` |
| **Token 概念** | 大模型按 Token 计费，1 个中文字 ≈ 1-2 个 Token |
| **Prompt 构造** | `system`/`user`/`assistant` 三种角色消息的含义 |
| **Temperature 参数** | 控制回答的"创意程度"，0=确定性最高，1=最随机 |

#### 怎么学

1. **第一步：** 注册 OpenAI 账号，获取 API Key，安装 `openai` 库
2. **最小闭环：** 用 5 行代码让 GPT 回答一个问题
3. **进阶练习：**
   - 写一个"多轮对话"程序（维护 message 历史）
   - 调用 Embedding API，把 3 句话变成向量，用余弦相似度比较哪两句最像
4. **理解 Embedding：** 把它想象成"语义 GPS 坐标"——意思相近的文本，在向量空间里距离近

#### 🎯 面试官最爱问

| 问题 | 考察点 |
|---|---|
| "什么是 Embedding？它和 One-Hot 编码有什么区别？" | 考察你是否理解**稠密向量 vs 稀疏向量**的本质差异。One-Hot 是"每个词一个独立维度"（稀疏、高维、无语义），Embedding 是"语义压缩"（稠密、低维、语义相近的词距离近）。 |
| "Temperature 设为 0 和设为 1，输出有什么区别？RAG 场景你会怎么选？" | RAG 场景一般选 **低温度（0-0.3）**，因为我们要的是"忠实于检索内容的回答"，不是创意写作。 |
| "Token 限制对你有什么影响？你怎么处理超长文档？" | 引出 **文本切分（Chunking）** 的必要性——这就是 RAG 的核心动机之一。 |

### 2.2 LangChain — 大模型的"乐高积木"

#### 核心学习知识点（20% 骨干）

| 概念 | 类比 | 说明 |
|---|---|---|
| **Model（模型）** | 发动机 | 对接 OpenAI 的封装层 |
| **Prompt Template（提示模板）** | 问卷模板 | 把变量填进预设的提示词里 |
| **Chain（链）** | 流水线 | 把多个步骤串起来：提示词 → 模型 → 输出解析 |
| **Document Loader（文档加载器）** | 送料机 | 把 PDF/TXT/网页等变成统一的 Document 对象 |
| **Text Splitter（文本分割器）** | 切纸机 | 把长文档切成小块（Chunk） |
| **Vector Store（向量存储）** | 智能书架 | 存储和检索向量化后的文本块 |
| **Retriever（检索器）** | 图书管理员 | 根据问题从书架上找到最相关的几本书 |
| **RetrievalQA Chain** | 完整流水线 | 检索 + 提问 + 生成回答的完整 RAG 链 |

#### 怎么学

1. **先跑通最简单的 Chain：**
   ```python
   from langchain_openai import ChatOpenAI
   from langchain_core.prompts import ChatPromptTemplate

   llm = ChatOpenAI(model="gpt-4o-mini")
   prompt = ChatPromptTemplate.from_template("用一句话解释：{concept}")
   chain = prompt | llm
   print(chain.invoke({"concept": "RAG"}))
   ```
2. **再跑通文档加载 → 切分 → 向量化 → 检索的完整链路**
3. **练习：** 用一个 PDF 文件，实现"问答机器人"

#### 🎯 面试官最爱问

| 问题 | 考察点 |
|---|---|
| "LangChain 的 LCEL（LangChain Expression Language）是什么？`\|` 操作符做了什么？" | LCEL 是 LangChain 的"管道语法"，`\|` 类似 Unix 的管道，把组件串联成 DAG（有向无环图），支持流式输出、异步调用、批处理。 |
| "LangChain 和直接调用 OpenAI API 相比，优势和劣势是什么？" | 优势：**抽象层**，换模型只需改一行代码；内置文档加载、切分、向量检索等工具链。劣势：**版本迭代太快**（v0.1→v0.2 大量 breaking changes）、调试困难、有时过度封装。面试时说出这些"坑"反而加分。 |
| "Document 对象长什么样？metadata 有什么用？" | `Document(page_content="文本", metadata={"source": "xxx.pdf", "page": 3})`。metadata 用于**来源追溯**——RAG 回答时可以告诉用户"答案来自哪个文件的第几页"。 |

### 2.3 FastAPI — 后端服务的"高速公路"

#### 核心学习知识点（20% 骨干）

| 概念 | 说明 |
|---|---|
| **路由装饰器** `@app.get()` / `@app.post()` | 定义"哪个 URL 对应哪个函数" |
| **Pydantic 数据校验** | 用 Python 类定义请求/响应的数据结构，自动校验 |
| **请求体（Request Body）** | POST 请求携带的 JSON 数据 |
| **依赖注入 `Depends()`** | FastAPI 的杀手锏——把公共逻辑（如数据库连接、认证）抽成可复用的依赖 |
| **异步 `async/await`** | 处理高并发的关键——等 API 响应时不阻塞其他请求 |
| **自动文档 Swagger UI** | `/docs` 路径自动生成 API 交互文档 |

#### 怎么学

1. **最小闭环：** 10 行代码启动一个 API 服务器
   ```python
   from fastapi import FastAPI
   app = FastAPI()

   @app.get("/")
   def read_root():
       return {"message": "Hello RAG!"}
   ```
2. **练习路线：**
   - 写一个 POST 接口，接收问题，返回固定回答
   - 加上 Pydantic 模型做请求校验
   - 把 RAG 链路接入接口，实现真正的问答 API
   - 加上异常处理、日志记录

#### 🎯 面试官最爱问

| 问题 | 考察点 |
|---|---|
| "FastAPI 的异步机制是怎么实现的？和 Flask 有什么区别？" | FastAPI 基于 **Starlette（ASGI 框架）** 和 **Pydantic**。ASGI 支持异步，WSGI（Flask）不支持。在 RAG 场景中，调用 OpenAI API 是 I/O 密集操作，异步可以让服务器同时处理多个请求而不阻塞。 |
| "Pydantic 的作用是什么？为什么不用字典？" | Pydantic 在**运行时强制校验数据类型**。字典不校验类型，传入错误数据会在代码深处才报错；Pydantic 在入口就拦住，并自动生成清晰的错误信息和 API 文档。 |
| "依赖注入 `Depends()` 解决了什么问题？" | 把"每次请求都需要做的公共逻辑"（如获取数据库连接、验证用户身份）抽成独立函数，避免重复代码，方便测试时替换依赖。 |

### 2.4 Chroma / Milvus — 向量的"智能图书馆"

#### 核心学习知识点（20% 骨干）

| 概念 | 类比 | 说明 |
|---|---|---|
| **Collection（集合）** | 书架 | 一组相关向量的容器 |
| **Embedding Function** | 分类系统 | 把文本变成向量的函数 |
| **添加文档** `collection.add()` | 上架 | 把文本 + 向量 + 元数据存进去 |
| **查询** `collection.query()` | 查找 | 给一个问题向量，返回最相似的 K 个结果 |
| **元数据过滤** | 按标签筛选 | `where={"source": "manual.pdf"}` |
| **持久化存储** | 图书馆存档 | Chroma 默认用 SQLite，Milvus 有自己的存储引擎 |

#### Chroma 快速上手

```python
import chromadb

client = chromadb.Client()  # 内存模式
collection = client.create_collection("my_docs")

collection.add(
    documents=["RAG是检索增强生成", "向量数据库存储Embedding"],
    ids=["doc1", "doc2"]
)

results = collection.query(query_texts=["什么是RAG？"], n_results=1)
print(results)  # 返回最相似的文档
```

#### Chroma vs Milvus 选择

| 维度 | Chroma | Milvus |
|---|---|---|
| 定位 | 轻量级、开发友好 | 生产级、高性能 |
| 数据量 | 十万级以内 | 百万到十亿级 |
| 部署 | 嵌入式 / Docker | Docker / K8s |
| 学习建议 | **先学 Chroma** | 项目成熟后迁移 |

#### 🎯 面试官最爱问

| 问题 | 考察点 |
|---|---|
| "向量检索和传统数据库的 LIKE 查询有什么本质区别？" | LIKE 是**字面匹配**（"猫"找不到"喵星人"），向量检索是**语义匹配**（通过 Embedding 的余弦相似度/欧氏距离，找到意思相近的文本）。这是 RAG 存在的根本原因。 |
| "什么是 ANN（近似最近邻）？为什么不直接算精确距离？" | 精确最近邻需要和**每一条向量**计算距离，时间复杂度 O(n)。百万级数据时太慢。ANN 用**索引结构**（如 HNSW 图、IVF 倒排索引）牺牲一小部分精度换取速度提升——类比"不翻遍整个图书馆，而是先查目录再定位书架"。 |
| "HNSW 是什么？简单描述原理。" | **分层可导航小世界图**——想象一个多层社交网络：顶层是"大人物"（少量节点，连接跨度大），底层是"普通人"（大量节点，连接紧密）。搜索时从顶层开始，逐层往下"跳"到最近的节点，像 GPS 导航从高速路 → 省道 → 小路逐步精确。 |

### 3.1 文本切分策略（Chunking）优化

#### 为什么重要？

切分是 RAG 的**地基**。切太大 → 检索不精准、塞不进 Token 限制；切太小 → 丢失上下文、回答碎片化。

#### 主流策略对比

| 策略 | 原理 | 适用场景 |
|---|---|---|
| **固定长度切分** | 按字符数/Token 数硬切 | 快速原型，不推荐生产 |
| **递归字符切分** `RecursiveCharacterTextSplitter` | 先按 `\n\n` 分段，段太长再按 `\n` 分，再按句号分… | **LangChain 默认推荐**，效果均衡 |
| **语义切分** | 用 Embedding 判断相邻句子的语义变化，在"话题跳转处"切 | 效果最好，但计算成本高 |
| **按文档结构切分** | 利用 Markdown 标题、HTML 标签、PDF 段落等天然结构 | 结构化文档（技术手册、法律文书） |

#### 优化要点

- **Chunk Size（块大小）：** 一般 500-1000 Token，需要实验调优
- **Chunk Overlap（重叠区域）：** 相邻块重叠 10%-20%，防止重要信息被切断在两块边界
- **Metadata 增强：** 切分时保留标题、章节等上下文信息，让每个 Chunk 自带"我来自哪里"

#### 面试加分话术

> "我尝试了固定长度和递归字符两种切分策略，发现递归字符在保持段落完整性上明显更好。我还加了 200 Token 的重叠区域，并在每个 Chunk 的 metadata 中注入了章节标题作为上下文前缀，检索召回率提升了约 15%。"

### 3.2 检索效果优化

#### 3.2.1 Hybrid Search（混合检索）

| 检索方式 | 原理 | 优势 | 劣势 |
|---|---|---|---|
| **稀疏检索（BM25）** | 基于关键词频率匹配 | 精确匹配术语时效果好 | 无法理解语义（"快乐"找不到"开心"） |
| **稠密检索（向量）** | 基于 Embedding 语义相似度 | 理解语义、同义词 | 可能忽略精确关键词 |
| **混合检索** | 同时用两种方式检索，合并结果 | **互补**，兼顾精确和语义 | 实现稍复杂 |

#### 混合检索的核心：分数融合

```
final_score = α × dense_score + (1-α) × sparse_score
```

α 是权重系数，一般从 0.5 开始调。这个公式叫 **Reciprocal Rank Fusion (RRF)** 或加权线性融合。

#### 3.2.2 Re-ranking（重排序）

| 阶段 | 做什么 | 为什么 |
|---|---|---|
| **第一阶段：召回** | 从向量库中快速检索 Top-K（如 K=20） | 快但粗糙 |
| **第二阶段：重排** | 用 Cross-Encoder 模型对 Top-K 精细打分 | 慢但精准 |
| **最终** | 取重排后的 Top-N（如 N=5）喂给 LLM | 送入 LLM 的都是最相关的内容 |

**常用重排模型：** Cohere Rerank API、BGE-Reranker、Cross-Encoder（`sentence-transformers`）

#### 面试加分话术

> "我的检索分三步：先用 BM25 和向量检索各取 Top-20，用 RRF 融合后取 Top-10，再用 BGE-Reranker 精排后取 Top-5 送入 LLM。这样既保证了召回率，又保证了精确度。"

### 3.3 RAG 系统评估与幻觉应对

#### 3.3.1 RAG 评估指标

| 指标 | 含义 | 怎么衡量 |
|---|---|---|
| **Context Precision（上下文精确率）** | 检索到的内容中有多少是真正相关的 | 检索质量 |
| **Context Recall（上下文召回率）** | 相关内容有多少被检索到了 | 检索覆盖度 |
| **Faithfulness（忠实度）** | 回答是否忠于检索到的上下文，没有"编造" | 幻觉检测 |
| **Answer Relevancy（答案相关性）** | 回答是否真正回答了用户的问题 | 回答质量 |

#### 3.3.2 Ragas 框架

```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision

result = evaluate(
    dataset=eval_dataset,  # 包含 question/answer/contexts/ground_truth
    metrics=[faithfulness, answer_relevancy, context_precision]
)
print(result)
```

#### 3.3.3 应对幻觉的实战策略

| 策略 | 原理 |
|---|---|
| **Prompt 约束** | 在 system prompt 中明确指令："只根据提供的上下文回答，如果上下文没有相关信息，请说'我无法根据现有资料回答'" |
| **来源引用** | 要求 LLM 在回答中标注来源（"根据文档第3节…"），便于用户验证 |
| **降低 Temperature** | 减少模型"发挥创造力"的空间 |
| **Relevance 阈值过滤** | 检索结果相似度低于阈值时，不送入 LLM，直接回复"未找到相关信息" |

#### 面试加分话术

> "我用 Ragas 框架对系统做了定量评估。初始版本的 Faithfulness 只有 0.72，分析发现主要是 Chunk 太小导致上下文不完整。把 Chunk Size 从 500 调到 800 并加入章节标题前缀后，Faithfulness 提升到 0.89。"
