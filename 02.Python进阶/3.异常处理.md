# 异常处理

> 调用 OpenAI API 会超时、读取文件会找不到、网络请求会失败——RAG 系统中充满了各种可能出错的地方。异常处理让你的程序**优雅地应对错误**，而不是直接崩溃。

---

## 1. 用"灭火器"来理解异常处理

程序运行就像日常生活：

- **正常情况**：做饭、看电视、工作
- **异常情况**：着火了、停电了、水管爆了

没有灭火器（异常处理）→ 着火了房子就烧了（程序崩溃）
有灭火器（异常处理）→ 着火了及时灭火，生活继续（程序继续运行）

## 2. 基本语法：try/except

```python
try:
    # 尝试执行的代码（可能出错）
    result = 10 / 0
except ZeroDivisionError:
    # 出错时执行的代码（捕获错误）
    print("不能除以零！")
```

### 完整语法

```python
try:
    # 可能出错的代码
    result = 10 / 0
except ZeroDivisionError as e:
    # 捕获特定异常，e 是错误信息
    print(f"出错了: {e}")
except Exception as e:
    # 捕获所有其他异常（兜底）
    print(f"未知错误: {e}")
else:
    # 没有出错时执行
    print(f"结果是: {result}")
finally:
    # 无论是否出错都会执行（清理资源）
    print("处理完毕")
```

## 3. 常见异常类型

| 异常类型 | 触发场景 | RAG 中的常见原因 |
|---------|---------|-----------------|
| `FileNotFoundError` | 文件不存在 | 知识库文件路径错误 |
| `KeyError` | 字典中 key 不存在 | API 响应格式不符合预期 |
| `IndexError` | 列表索引越界 | 检索结果为空时取第一个 |
| `TypeError` | 类型不匹配 | 把 None 当字符串拼接 |
| `ValueError` | 值不合法 | 输入了非法参数 |
| `ConnectionError` | 网络连接失败 | 调用 API 时网络不通 |
| `TimeoutError` | 操作超时 | API 响应太慢 |
| `PermissionError` | 权限不足 | 没有文件读写权限 |
| `ImportError` | 模块导入失败 | 缺少依赖包 |
| `AttributeError` | 对象没有该属性 | 调用了不存在的方法 |

```python
# 逐一演示

# FileNotFoundError
try:
    with open("不存在的文件.txt") as f:
        content = f.read()
except FileNotFoundError:
    print("文件不存在")

# KeyError
try:
    data = {"name": "RAG"}
    value = data["age"]
except KeyError as e:
    print(f"键 {e} 不存在")

# IndexError
try:
    items = []
    first = items[0]
except IndexError:
    print("列表为空，无法取第一个元素")

# TypeError
try:
    result = "分数: " + 0.95
except TypeError:
    print("字符串不能和数字直接拼接，需要用 str() 转换")
```

## 4. 自定义异常

当内置异常不能准确描述错误时，可以自定义异常类：

```python
class APIError(Exception):
    """API 调用异常"""
    def __init__(self, message, status_code=None):
        super().__init__(message)
        self.status_code = status_code

class RateLimitError(APIError):
    """API 限流异常"""
    pass

# 使用自定义异常
def call_api(query):
    # 模拟 API 调用
    raise RateLimitError("请求太频繁，请稍后重试", status_code=429)

try:
    call_api("什么是RAG？")
except RateLimitError as e:
    print(f"限流: {e}，状态码: {e.status_code}")
except APIError as e:
    print(f"API 错误: {e}")
```

## 5. 异常处理的最佳实践

### 5.1 只捕获你能处理的异常

```python
# 不好的写法：捕获所有异常但什么都不做（吞掉错误）
try:
    result = risky_operation()
except:
    pass                # 错误被隐藏了，调试时找不到问题

# 好的写法：捕获特定异常，做有意义的处理
try:
    result = risky_operation()
except FileNotFoundError:
    result = default_value
    print("使用默认值")
except ConnectionError:
    print("网络不通，稍后重试")
```

### 5.2 用 finally 做清理工作

```python
file = None
try:
    file = open("data.txt", "r")
    content = file.read()
except FileNotFoundError:
    print("文件不存在")
finally:
    if file:
        file.close()    # 确保文件被关闭
```

### 5.3 记录错误日志

```python
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

try:
    result = risky_operation()
except Exception as e:
    logger.error(f"操作失败: {e}", exc_info=True)
    # exc_info=True 会记录完整的错误堆栈
```

## 6. 重试机制（RAG 中非常重要）

调用 API 时，经常因为网络波动或限流导致临时失败，这时候**自动重试**是最佳策略：

```python
import time

def call_with_retry(func, max_retries=3, delay=1):
    """
    带重试的函数调用
    - func: 要执行的函数
    - max_retries: 最大重试次数
    - delay: 每次重试间隔（秒）
    """
    for attempt in range(max_retries):
        try:
            return func()              # 尝试执行
        except ConnectionError as e:
            if attempt < max_retries - 1:
                wait = delay * (2 ** attempt)    # 指数退避：1s, 2s, 4s...
                print(f"第 {attempt + 1} 次失败，{wait}秒后重试: {e}")
                time.sleep(wait)
            else:
                print(f"重试 {max_retries} 次后仍然失败")
                raise                # 把异常抛出去
```

### 在 RAG 中的使用

```python
# 模拟一个不稳定的 API
import random

def unreliable_api():
    """模拟一个 50% 概率失败的 API"""
    if random.random() < 0.5:
        raise ConnectionError("API 超时")
    return "API 返回的结果"

# 使用重试机制
try:
    result = call_with_retry(unreliable_api, max_retries=3, delay=1)
    print(f"成功: {result}")
except ConnectionError:
    print("所有重试都失败了，使用备用方案")
```

## 7. 在 RAG 项目中的应用场景

| 异常处理场景 | RAG 中的具体用途 |
|------------|-----------------|
| 文件读取 | 知识库文件不存在或编码错误时，跳过该文件继续处理 |
| API 调用 | OpenAI API 超时/限流时自动重试 |
| JSON 解析 | API 返回格式异常时，记录错误并使用默认值 |
| 向量检索 | 查询向量维度不匹配时，给出明确错误提示 |
| 网络请求 | 网络断开时等待重连，而不是程序崩溃 |
| 批量处理 | 处理 1000 个文档时，个别文件出错不影响整体进度 |

### RAG 管道中的异常处理模式

```python
def process_documents(file_paths):
    """处理多个文档，单个失败不影响整体"""
    results = []
    errors = []

    for path in file_paths:
        try:
            with open(path, "r", encoding="utf-8") as f:
                content = f.read()
            results.append({"path": path, "content": content})
        except FileNotFoundError:
            errors.append({"path": path, "error": "文件不存在"})
        except UnicodeDecodeError:
            errors.append({"path": path, "error": "编码错误"})
        except Exception as e:
            errors.append({"path": path, "error": str(e)})

    print(f"成功: {len(results)} 个，失败: {len(errors)} 个")
    return results, errors
```

## 8. 面试官常问问题

**Q1: `try/except` 和 `if/else` 有什么区别？什么时候用哪个？**

A: `if/else` 是**预防性**检查（已知可能的问题），`try/except` 是**兜底**处理（无法提前预知的错误）。比如读文件前先用 `os.path.exists()` 检查（if/else），但网络请求只能用 try/except 因为你无法预知网络状态。

**Q2: 为什么不建议用 `except Exception` 捕获所有异常？**

A: 因为它会隐藏真正的 Bug。比如代码有 `NameError`（变量名写错），被 `except Exception` 捕获后你永远发现不了这个 Bug。应该只捕获你**预期会发生的**特定异常。

**Q3: `finally` 有什么用？什么时候需要它？**

A: `finally` 无论是否出错都会执行，用于**清理资源**：关闭文件、释放数据库连接、删除临时文件等。`with` 语句本质上就是 `try/finally` 的语法糖。

**Q4: RAG 项目中调用 OpenAI API 时，常见的异常有哪些？怎么处理？**

A: `openai.RateLimitError`（限流，等待后重试）、`openai.APITimeoutError`（超时，重试）、`openai.APIConnectionError`（网络问题，检查网络后重试）、`openai.AuthenticationError`（API Key 错误，检查 Key）。通常用重试机制 + 指数退避处理。

**Q5: 什么是"指数退避"重试策略？**

A: 每次重试的等待时间翻倍：1秒、2秒、4秒、8秒...这样可以避免在服务端繁忙时反复发送请求加重负担，是调用外部 API 的标准做法。
