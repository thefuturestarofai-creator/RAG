# 包管理与虚拟环境

> 你写的 RAG 项目依赖 `langchain`、`openai`、`chromadb` 等第三方库。如何安装、管理、锁定这些依赖的版本？如何避免不同项目之间的依赖冲突？这就是包管理和虚拟环境要解决的问题。

---

## 1. 用"药箱"来理解虚拟环境

想象你在管理一个**药箱**：

- **全局环境**（没有虚拟环境）= 所有药都放在一个大箱子里。感冒药和胃药混在一起，找起来麻烦，还可能拿错。
- **虚拟环境** = 每个项目有自己的**专属小药箱**。项目 A 需要 `langchain 0.1`，项目 B 需要 `langchain 0.2`，各自独立，互不干扰。

## 2. pip：Python 的包管理器

`pip` 是 Python 安装第三方库的工具，就像手机的应用商店。

### 2.1 基本命令

```bash
# 安装包
pip install langchain              # 安装最新版
pip install langchain==0.1.0       # 安装指定版本
pip install langchain>=0.1.0       # 安装 0.1.0 及以上版本

# 升级包
pip install --upgrade langchain    # 升级到最新版

# 卸载包
pip uninstall langchain

# 查看已安装的包
pip list

# 查看某个包的信息
pip show langchain

# 搜索包
pip search langchain               # （有时会被禁用，直接去 pypi.org 搜）
```

### 2.2 使用国内镜像源（加速下载）

国内访问 PyPI 官方源很慢，可以换用国内镜像：

```bash
# 临时使用
pip install langchain -i https://pypi.tuna.tsinghua.edu.cn/simple

# 永久设置（推荐）
pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple
```

常用镜像源：
| 镜像 | 地址 |
|------|------|
| 清华 | https://pypi.tuna.tsinghua.edu.cn/simple |
| 阿里云 | https://mirrors.aliyun.com/pypi/simple |
| 豆瓣 | https://pypi.douban.com/simple |

## 3. requirements.txt：锁定依赖版本

### 3.1 什么是 requirements.txt？

它是一个**清单文件**，列出了项目需要的所有第三方包及其版本。别人拿到你的项目后，一条命令就能安装所有依赖。

```
# requirements.txt 示例
langchain==0.1.0
openai==1.12.0
chromadb==0.4.22
tiktoken==0.5.2
```

### 3.2 生成和使用

```bash
# 从当前环境导出所有依赖
pip freeze > requirements.txt

# 从 requirements.txt 安装所有依赖
pip install -r requirements.txt
```

### 3.3 最佳实践

```
# requirements.txt 中的版本号写法
langchain==0.1.0       # 精确版本（推荐用于生产环境）
openai>=1.0,<2.0       # 范围版本（兼容性更好）
chromadb               # 不指定版本（不推荐，可能安装到不兼容的版本）
```

## 4. venv：Python 内置的虚拟环境

### 4.1 创建虚拟环境

```bash
# 在项目目录下创建虚拟环境
python -m venv venv

# 命名规则：
# - 通常叫 venv 或 .venv
# - 会在当前目录下创建一个 venv 文件夹
# - 这个文件夹包含了独立的 Python 解释器和包
```

### 4.2 激活虚拟环境

```bash
# Windows (CMD)
venv\Scripts\activate.bat

# Windows (PowerShell)
venv\Scripts\Activate.ps1

# macOS / Linux
source venv/bin/activate

# 激活后，命令行前面会出现 (venv) 标记：
# (venv) C:\Users\你\project>
```

### 4.3 退出虚拟环境

```bash
deactivate
```

### 4.4 完整工作流程

```bash
# 1. 创建项目目录
mkdir my_rag_project
cd my_rag_project

# 2. 创建虚拟环境
python -m venv venv

# 3. 激活虚拟环境
venv\Scripts\activate          # Windows

# 4. 安装依赖
pip install langchain openai chromadb

# 5. 导出依赖列表
pip freeze > requirements.txt

# 6. 写代码、运行...

# 7. 退出虚拟环境
deactivate
```

## 5. 常见工具对比

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| `pip + venv` | Python 内置，简单直接 | 入门学习、小项目 |
| `conda` | 跨语言包管理，支持非 Python 包 | 数据科学、需要特定 C 库 |
| `poetry` | 现代化包管理，自动处理依赖 | 正式项目开发 |
| `pipenv` | pip + venv 的结合体 | 中等规模项目 |

**面试回答建议**：入门用 `pip + venv` 就够了，了解 `poetry` 是加分项。

## 6. 为什么要用虚拟环境？

### 问题场景

```
项目 A 需要 langchain==0.1.0
项目 B 需要 langchain==0.2.0

如果不用虚拟环境：
pip install langchain==0.1.0   → 安装了 0.1.0
pip install langchain==0.2.0   → 覆盖成了 0.2.0
项目 A 就跑不了了！
```

### 虚拟环境的解决方案

```
项目 A/
├── venv/               → 独立环境，langchain==0.1.0
└── main.py

项目 B/
├── venv/               → 独立环境，langchain==0.2.0
└── main.py

两个项目互不影响！
```

## 7. .gitignore 文件

虚拟环境文件夹**不应该**提交到 Git：

```gitignore
# .gitignore
venv/                   # 忽略虚拟环境
__pycache__/            # 忽略 Python 缓存
*.pyc                   # 忽略编译文件
.env                    # 忽略环境变量文件（可能包含 API Key）
```

**为什么？**
- venv 文件夹可能有几百 MB，上传/下载都慢
- `requirements.txt` 已经记录了所有依赖，别人可以自己重建
- `.env` 可能包含 API Key，泄露会有安全风险

---

## 8. 环境变量与 .env 文件

API Key 等敏感信息不应该写在代码里，应该放在环境变量中：

```python
# 不好的写法（API Key 暴露在代码中）
api_key = "sk-xxxxxxxxxxxx"

# 好的写法（从环境变量读取）
import os
api_key = os.environ.get("OPENAI_API_KEY")

# 或者用 python-dotenv 从 .env 文件读取
from dotenv import load_dotenv
load_dotenv()
api_key = os.environ.get("OPENAI_API_KEY")
```

```
# .env 文件（不要提交到 Git！）
OPENAI_API_KEY=sk-xxxxxxxxxxxx
DATABASE_URL=postgresql://localhost/mydb
```

## 9. 在 RAG 项目中的应用场景

| 包管理操作 | RAG 中的具体用途 |
|----------|-----------------|
| `pip install langchain` | 安装 RAG 框架 |

| `pip install chromadb` | 安装向量数据库 |

## 10. 面试官常问问题

**Q1: 什么是虚拟环境？为什么需要它？**

A: 虚拟环境是 Python 的**独立运行空间**，每个项目有自己的包和版本，互不干扰。没有虚拟环境的话，不同项目依赖同一个包的不同版本会冲突。

**Q2: `pip install` 和 `conda install` 有什么区别？**

A: `pip` 只能安装 Python 包，从 PyPI 下载；`conda` 可以安装 Python 包和非 Python 依赖（如 C 库），有自己的包仓库。RAG 项目用 `pip` 就够了。

**Q3: `requirements.txt` 有什么用？为什么不直接 `pip install`？**

A: `requirements.txt` 记录了项目的所有依赖和精确版本，保证**任何人、任何时候**都能复现相同的环境。直接 `pip install` 可能装到不同版本，导致"在我电脑上能跑"的问题。

**Q4: 如果 `pip install` 很慢怎么办？**

A: 使用国内镜像源（清华、阿里云）。`pip config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple`。

**Q5: 为什么 `.env` 文件不能提交到 Git？**

A: `.env` 通常包含 API Key、数据库密码等敏感信息。提交到 Git（尤其是公开仓库）会导致密钥泄露，被他人滥用。应该用 `.gitignore` 忽略它，并在 README 中说明如何创建 `.env`。
