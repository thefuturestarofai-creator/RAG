# 重排序 Re-ranking

## 核心概念

重排序就像"海选→复试"的两阶段流程：
- **第一阶段（召回）**：快速从海量文档中筛选出候选集（海选）
- **第二阶段（重排）**：用更精细的模型对候选集重新排序（复试）

### 类比理解

想象招聘流程：
- **召回阶段**：HR 快速筛选简历，从 1000 份中选出 50 份进入面试（速度快，精度一般）
- **重排阶段**：技术面试官仔细评估这 50 人，重新排名（速度慢，精度高）

## 两阶段检索架构

### 第一阶段：召回（Retrieval）

**目标**：快速找到相关文档的候选集

**方法**：
- BM25 稀疏检索
- 向量稠密检索
- 混合检索

**特点**：
- 速度快（毫秒级）
- 召回率高（尽量不漏）
- 精度一般（可能有噪声）

**输出**：Top-K 个候选文档（如 K=50）

### 第二阶段：重排（Re-ranking）

**目标**：对候选集精细排序，提高精度

**方法**：
- Cross-Encoder 模型
- 专业 Rerank 模型（Cohere、BGE-Reranker）

**特点**：
- 速度慢（需要逐对计算）
- 精度高（深度理解 query-doc 关系）

**输出**：Top-N 个最终文档（如 N=5）

## Cross-Encoder 原理

### 与 Bi-Encoder 的区别

**Bi-Encoder（双塔模型）**：
```
Query → Encoder → Vector Q
Doc   → Encoder → Vector D
相似度 = cosine(Q, D)
```
- Query 和 Doc 独立编码
- 可以预计算文档向量
- 速度快，适合召回

**Cross-Encoder（交叉编码器）**：
```
[Query, Doc] → Encoder → 相似度分数
```
- Query 和 Doc 拼接后一起编码
- 能捕捉 query-doc 之间的细粒度交互
- 速度慢，适合重排

### Cross-Encoder 工作流程

```python
# 伪代码
def cross_encoder_score(query, document):
    # 1. 拼接 query 和 document
    input_text = f"[CLS] {query} [SEP] {document} [SEP]"

    # 2. 送入 BERT 等预训练模型
    outputs = bert_model(input_text)

    # 3. 取 [CLS] token 的表示
    cls_embedding = outputs.last_hidden_state[:, 0, :]

    # 4. 通过分类头得到相似度分数
    score = linear_layer(cls_embedding)

    return score
```

### 为什么 Cross-Encoder 更准？

1. **注意力机制**：Query 和 Doc 的 token 之间可以相互关注
2. **细粒度交互**：能捕捉词级别的匹配关系
3. **预训练知识**：利用 BERT 等模型的语言理解能力

## 主流 Rerank 模型

### Cohere Rerank

**特点**：
- 商业 API，开箱即用
- 支持多语言
- 效果好，速度快

**使用方式**：
```python
import cohere

co = cohere.Client(API_KEY)
results = co.rerank(
    query="什么是机器学习？",
    documents=["机器学习是...", "深度学习是..."],
    top_n=5
)
```

### BGE-Reranker

**特点**：
- 开源免费
- 中文效果好
- 可本地部署

**模型系列**：
- `bge-reranker-base`：基础版
- `bge-reranker-large`：大模型版
- `bge-reranker-v2-m3`：多语言版

**使用方式**：
```python
from sentence_transformers import CrossEncoder

model = CrossEncoder('BAAI/bge-reranker-base')
pairs = [["query", "doc1"], ["query", "doc2"]]
scores = model.predict(pairs)
```

### bge-reranker-v2-m3

**特点**：
- 支持中英文
- 支持长文本（最长 8192 tokens）
- MTEB Reranking 排行榜表现优异

## 面试加分话术

### 问题：为什么需要两阶段检索？

**回答**：
"这是效率和精度的权衡。如果直接用 Cross-Encoder 对百万文档做重排，计算成本太高。所以先用 BM25/向量检索快速召回 Top-50，再用 Cross-Encoder 精排到 Top-5。就像招聘先筛简历再面试，既保证效率又保证质量。"

### 问题：Cross-Encoder 和 Bi-Encoder 有什么区别？

**回答**：
"核心区别在于 Query 和 Doc 是否交互。Bi-Encoder 独立编码，速度快但精度有限；Cross-Encoder 把 Query 和 Doc 拼接一起编码，能捕捉细粒度交互，精度更高但速度慢。实践中，Bi-Encoder 用于召回，Cross-Encoder 用于重排。"

### 问题：Rerank 模型怎么选？

**回答**：
"看场景。如果是英文为主且预算充足，Cohere Rerank 效果最好。如果需要中文支持或本地部署，推荐 BGE-Reranker。如果需要处理长文本，选 bge-reranker-v2-m3，它支持 8192 tokens。"

## 常见追问

**Q: 重排会增加多少延迟？**

A: 取决于候选文档数量和文档长度。假设 50 个候选文档，平均 512 tokens，用 BGE-Reranker-Base 在 GPU 上大约 50-100ms。可以接受。

**Q: 重排的 Top-K 取多少合适？**

A: 召回阶段 Top-K 通常取 20-100，重排后 Top-N 通常取 3-10。具体取决于：
- 下游任务（问答取 3-5，摘要取 5-10）
- Token 限制（LLM 上下文窗口）
- 精度要求

**Q: 能不能只用 Cross-Encoder 不用召回？**

A: 理论上可以，但实践中不可行。假设 100 万文档，每个 512 tokens，Cross-Encoder 处理需要数小时。所以必须先召回缩小范围。

**Q: Rerank 和 Fine-tuning 怎么选？**

A: 两者可以结合：
- Rerank：不需要训练数据，开箱即用
- Fine-tuning：需要标注数据，效果更好
- 实践中：先用 Rerank 验证方案，再考虑 Fine-tuning 提升
