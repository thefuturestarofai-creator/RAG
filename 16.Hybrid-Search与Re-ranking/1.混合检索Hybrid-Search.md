# 混合检索 Hybrid Search

## 核心概念

混合检索就像"双引擎搜索"——同时启动两个搜索引擎，一个擅长精确匹配，一个擅长语义理解，最后把两个引擎的结果合并排序。

### 类比理解

想象你在图书馆找书：
- **稀疏检索（BM25）**：像图书管理员按书名、关键词精确查找，"你说要找《机器学习》，我就找书名里有这几个字的"
- **稠密检索（向量）**：像一位理解你意图的顾问，"你想学AI，我推荐《统计学习方法》《深度学习》这些内容相关的书"

混合检索 = 两个都用，最后综合排序

## 两种检索方式对比

### 稀疏检索（Sparse Retrieval）

**代表算法**：BM25（Best Matching 25）

**原理**：
- 基于词频（TF）和逆文档频率（IDF）
- 计算查询词在文档中的出现频率和稀有程度
- 本质是"关键词匹配"

**优点**：
- 精确匹配效果好
- 不需要训练
- 速度快，资源消耗低
- 对专有名词、数字、缩写效果好

**缺点**：
- 无法理解语义相似性
- "电脑"和"计算机"被认为是不同词
- 需要关键词完全匹配

### 稠密检索（Dense Retrieval）

**代表算法**：向量检索（如 text-embedding-ada-002）

**原理**：
- 将文本转换为高维向量
- 通过余弦相似度等计算向量距离
- 本质是"语义匹配"

**优点**：
- 理解语义相似性
- "电脑"和"计算机"会被认为相似
- 能处理同义词、近义词

**缺点**：
- 对精确匹配不敏感
- 可能返回语义相关但不精确的结果
- 需要 embedding 模型，资源消耗大

## 分数融合公式

### 方法一：加权线性融合（Weighted Linear Combination）

最直观的融合方式，直接对两种检索的分数加权求和：

```
final_score = α × dense_score + (1-α) × sparse_score
```

其中：
- `dense_score`：向量检索的相似度分数（0-1）
- `sparse_score`：BM25 的归一化分数（0-1）
- `α`：权重参数（0-1），控制两种检索的贡献比例
  - α > 0.5：更重视语义匹配
  - α < 0.5：更重视关键词匹配
  - α = 0.5：两者等权重

**优点**：直观、可控、易于调参
**缺点**：需要对不同检索器的分数做归一化

### 方法二：RRF（Reciprocal Rank Fusion）

RRF 是一种基于排名的融合方法，不关心具体分数，只关心排名：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

其中：
- `d` 是文档
- `k` 是常数（通常取 60）
- `rank_i(d)` 是文档 d 在第 i 个检索器中的排名

**举例说明**：

假设文档 D 在两个检索器中的排名：
- BM25 排名：第 2 名
- 向量检索排名：第 5 名
- k = 60

RRF 分数 = 1/(60+2) + 1/(60+5) = 1/62 + 1/65 ≈ 0.0161 + 0.0154 = 0.0315

### 为什么用 RRF？

1. **简单**：不需要复杂的权重调参
2. **鲁棒**：对不同检索器的分数尺度不敏感
3. **有效**：实践中效果很好
4. **可扩展**：可以融合多个检索器

### 两种方法对比

| 对比维度 | 加权线性融合 | RRF |
|----------|-------------|-----|
| 融合对象 | 分数 | 排名 |
| 需要归一化 | 是 | 否 |
| 参数调优 | α 权重 | k 常数 |
| 适用场景 | 分数尺度相近 | 分数尺度差异大 |
| 实现复杂度 | 低 | 低 |

## 面试加分话术

### 问题：为什么需要混合检索？

**回答**：
"混合检索的核心是互补。BM25 擅长精确匹配，比如用户搜索'iPhone 15 Pro Max 256GB'，BM25 能精准找到包含这些关键词的文档。而向量检索擅长语义理解，比如用户问'苹果最新旗舰手机'，向量检索能找到语义相关的文档。在实际业务中，用户查询往往是两者的混合，所以混合检索能覆盖更多场景。"

### 问题：RRF 是怎么工作的？

**回答**：
"RRF 的核心思想是'排名融合'。它不关心每个检索器给出的具体分数，只关心排名。公式是 1/(k+rank)，k 通常取 60 是为了让排名靠后的文档分数衰减更快。这样即使两个检索器的分数尺度完全不同，也能公平地融合。"

### 问题：混合检索的权重怎么调？

**回答**：
"实践中，我会先用 RRF 做无权重融合，效果通常就不错。如果需要调优，可以用开发集做网格搜索，找到最优的 alpha 参数。也可以根据业务特点调整，比如搜索专有名词多的场景，可以给 BM25 更高权重。"

### 核心话术：完整检索流程

**问题：你的检索策略是怎样的？**

**回答**：
"我的检索分三步：先用BM25和向量检索各取Top-20，用RRF融合后取Top-10，再用BGE-Reranker精排后取Top-5送入LLM。"

**详细版**：
"在检索策略上，我采用了混合检索 + Re-ranking 的两阶段架构：

第一阶段：混合检索（召回）
- BM25 稀疏检索：擅长精确匹配，对专有名词效果好，取 Top-20
- 向量稠密检索：擅长语义理解，对同义词效果好，取 Top-20
- RRF 融合：将两种检索结果用 RRF 算法融合，取 Top-10

第二阶段：Re-ranking（精排）
- 使用 BGE-Reranker 对 Top-10 候选文档重排序
- Cross-Encoder 能捕捉 query-doc 之间的细粒度交互
- 最终取 Top-5 送入 LLM 生成回答

这样既保证了召回率（不遗漏），又保证了精确率（少噪声）。"

## 常见追问

**Q: BM25 和 TF-IDF 有什么区别？**

A: BM25 是 TF-IDF 的改进版。主要改进：
1. 引入文档长度归一化
2. 词频饱和机制（词频很高时，分数增长变慢）
3. 参数可调（b 控制长度归一化，k1 控制词频饱和）

**Q: 向量检索的 embedding 模型怎么选？**

A: 考虑因素：
1. 任务类型：通用 vs 领域特定
2. 维度：高维更精准但更慢
3. 语言支持：中文场景选中文模型
4. 性能：MTEB 排行榜参考

**Q: 混合检索会增加多少延迟？**

A: 两种检索可以并行执行，所以延迟约等于最慢的那个。向量检索通常比 BM25 慢，但通过 ANN 索引（如 HNSW）可以控制在毫秒级。总延迟增加通常 < 50ms。
