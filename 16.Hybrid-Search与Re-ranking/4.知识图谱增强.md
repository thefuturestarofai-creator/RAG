## 一、什么是知识图谱？

### 1.1 用"社交网络"类比

想象一个社交网络：

```
        朋友          同事
张三 ------→ 李四 ------→ 王五
 |                         |
 | 师生                    | 上下级
 ↓                         ↓
赵六（教授）              钱七（CEO）
```

知识图谱就是这样的**网络结构**，由两个核心元素组成：

- **实体（Entity）**：图中的节点，如"张三"、"李四"、"公司A"
- **关系（Relation）**：节点之间的连线，如"朋友"、"同事"、"任职于"

用数学语言说，知识图谱是一个**有向图 G = (E, R)**，其中：
- E 是实体集合
- R 是关系集合
- 每条边表示为三元组：**(头实体, 关系, 尾实体)**

例如：
```
(张三, 朋友, 李四)
(李四, 同事, 王五)
(王五, 任职于, 公司A)
(公司A, CEO, 钱七)
```

### 1.2 知识图谱 vs 传统数据库

| 对比维度 | 传统数据库 | 知识图谱 |
|----------|-----------|---------|
| 数据结构 | 表格（行和列） | 图（节点和边） |
| 查询方式 | SQL 查询 | 图遍历/路径查询 |
| 擅长场景 | 精确查询 | 关系推理 |
| 灵活性 | 加字段需要改表 | 随时加新关系 |

## 二、什么是 Graph RAG？

### 2.1 传统 RAG 的局限

传统 RAG 使用**向量检索**，擅长找"语义相似"的内容：

```
用户问："张三的上司是谁？"

传统 RAG 的做法：
1. 将"张三的上司是谁"转为向量
2. 在知识库中找语义最相似的文档片段
3. 希望那个片段里恰好提到了张三的上司

问题：如果信息分散在多个文档中，单靠语义相似度很难找到完整答案。
```

### 2.2 Graph RAG 的思路

Graph RAG = **向量检索** + **知识图谱的结构化关系检索**

```
用户问："张三的上司是谁？"

Graph RAG 的做法：
1. 向量检索：找到与"张三"相关的文档片段
2. 图检索：在知识图谱中找到"张三"这个实体
3. 图遍历：沿着"上司"关系走一步 → 找到"李四"
4. 融合：将向量检索结果和图检索结果合并，交给 LLM 生成答案
```

### 2.3 核心区别

```
向量检索：找"语义相似"  →  "张三的上司" ≈ "张三的领导是谁" ✓
图检索：  找"关系链路"  →  张三 →[上司]→ 李四 ✓（精确的关系路径）

两者结合 = 既理解语义，又理解关系
```

## 三、Graph RAG vs 传统 RAG 对比

| 对比维度 | 传统 RAG | Graph RAG |
|----------|---------|-----------|
| 检索方式 | 向量相似度 | 向量 + 图遍历 |
| 擅长场景 | 语义匹配 | 关系推理、多跳查询 |
| 信息组织 | 文档片段 | 实体-关系网络 |
| 多跳推理 | 困难 | 天然支持 |
| 实现复杂度 | 较低 | 较高 |
| 构建成本 | 低（直接切分文档） | 高（需要抽取实体关系） |

## 四、典型场景：多跳推理

### 4.1 什么是多跳推理？

多跳推理 = 需要沿着关系链走多步才能得到答案。

```
示例："张三的上司的手机号是多少？"

第一跳：张三 →[上司]→ 李四
第二跳：李四 →[手机号]→ 138xxxx1234

答案：138xxxx1234
```

### 4.2 传统 RAG 为什么做不好？

```
传统 RAG 的困境：
- 检索"张三的上司的手机号" → 语义相似的文档可能没有直接答案
- 需要先找到"张三的上司是李四"，再找到"李四的手机号"
- 这两个信息可能在不同文档中，向量检索很难一次性找到

Graph RAG 的优势：
- 直接在图上走两步：张三 → 李四 → 138xxxx1234
- 精确、高效、不依赖语义相似度
```

### 4.3 更多多跳场景

```
场景1："哪个供应商供应了苹果手机的屏幕？"
  苹果手机 →[使用组件]→ 屏幕 →[供应商]→ 供应商A

场景2："张三所在部门的总监是谁？"
  张三 →[所属部门]→ 技术部 →[总监]→ 王总监

场景3："这本书的作者还写过哪些书？"
  这本书 →[作者]→ 作者A →[著作]→ [书1, 书2, 书3]
```

## 五、简化实现思路

### 5.1 整体流程

```
阶段1：知识图谱构建
  原始文档 → LLM抽取实体和关系 → 三元组 → 构建图

阶段2：检索融合
  用户提问 → 向量检索 + 图检索 → 结果融合 → LLM生成答案
```

### 5.2 阶段1：实体关系抽取

用 LLM 从文本中抽取实体和关系：

```
输入文本：
"张三是技术部的高级工程师，他的直属上司是李四。
 李四是技术部的总监，手机号是 13812345678。"

LLM 抽取结果：
实体：张三、李四、技术部
关系：
  (张三, 职位, 高级工程师)
  (张三, 所属部门, 技术部)
  (张三, 直属上司, 李四)
  (李四, 职位, 总监)
  (李四, 所属部门, 技术部)
  (李四, 手机号, 13812345678)
```

### 5.3 阶段2：图检索融合

```
用户问："张三的上司的手机号是多少？"

步骤1：实体识别 → 识别出"张三"
步骤2：意图分析 → 需要走"上司"→"手机号"两步
步骤3：图遍历 → 张三 →[上司]→ 李四 →[手机号]→ 13812345678
步骤4：向量检索 → 找到相关文档片段（辅助验证）
步骤5：融合结果 → 交给 LLM 生成最终答案
```

### 5.4 用 NetworkX 构建知识图谱

```python
import networkx as nx

# 创建知识图谱
G = nx.DiGraph()

# 添加三元组（实体1, 实体2, 关系）
G.add_edge("张三", "李四", relation="直属上司")
G.add_edge("李四", "13812345678", relation="手机号")
G.add_edge("张三", "技术部", relation="所属部门")

# 查询：张三的上司的手机号
path = nx.shortest_path(G, "张三", "13812345678")
# 结果：['张三', '李四', '13812345678']
```

## 六、面试加分话术

### 话术模板

> "传统 RAG 使用向量检索，擅长语义匹配但不擅长关系推理。Graph RAG 在此基础上引入知识图谱，通过实体-关系图来补充结构化检索能力。核心流程是：用 LLM 从文档中抽取实体和关系构建知识图谱，查询时结合向量检索和图遍历，将两种结果融合后交给 LLM 生成答案。Graph RAG 最大的优势是支持多跳推理，比如'张三的上司的手机号'这种需要沿关系链走多步的问题，传统 RAG 很难处理，但 Graph RAG 可以通过图遍历精确找到答案。实现上可以用 NetworkX 构建图，用 LLM 做实体关系抽取。"

### 关键词

- **知识图谱（Knowledge Graph）**
- **实体-关系三元组（Entity-Relation Triple）**
- **多跳推理（Multi-hop Reasoning）**
- **向量检索 + 图检索融合**
- **实体关系抽取（Entity/Relation Extraction）**

## 七、面试常问问题

### Q1: 什么是 Graph RAG？和传统 RAG 有什么区别？

**参考答案**：
> Graph RAG 是在传统向量检索 RAG 基础上，引入知识图谱的结构化关系检索。传统 RAG 使用向量相似度找语义相近的文档片段，擅长语义匹配但不擅长关系推理。Graph RAG 通过构建实体-关系图，支持图遍历查询，能够处理多跳推理问题。例如"张三的上司的手机号"需要沿关系链走两步，传统 RAG 很难处理，Graph RAG 可以通过图遍历精确找到。实际实现中，通常将向量检索和图检索的结果融合，交给 LLM 生成最终答案。

### Q2: 知识图谱是怎么构建的？

**参考答案**：
> 知识图谱的构建主要有两种方式。第一种是用 LLM 从非结构化文本中抽取实体和关系，形成三元组（头实体, 关系, 尾实体），然后导入图数据库。第二种是从已有的结构化数据（如数据库表格）转换而来。实际项目中，通常先用 LLM 做实体关系抽取，再人工审核校正，最后存入图数据库（如 Neo4j）或用 NetworkX 等库管理。抽取质量直接影响 Graph RAG 的效果，所以这一步需要仔细设计 Prompt 和校验流程。

### Q3: Graph RAG 适合什么场景？什么时候用传统 RAG 就够了？

**参考答案**：
> Graph RAG 适合需要关系推理和多跳查询的场景，比如企业内部的知识问答（员工关系、组织架构）、供应链查询（哪个供应商供应了哪个产品的哪个部件）、学术文献查询（作者合作关系、引用链路）。如果只是简单的文档问答，用户的问题主要是语义匹配类的（如"什么是 RAG"），传统 RAG 就足够了。Graph RAG 的构建成本较高（需要抽取实体关系、维护图结构），只有在关系推理是核心需求时才值得投入。
