# Query 改写

## 核心概念

Query 改写就像"翻译官"——把用户模糊、口语化的问题转换成更适合检索的形式。

### 类比理解

想象你问图书馆管理员：
- **你说**："我想学点AI的东西"
- **管理员理解**：你是想找机器学习、深度学习、神经网络相关的书籍
- **管理员行动**：用多个关键词帮你搜索

Query 改写就是让 LLM 充当这个"管理员"。

## 为什么需要 Query 改写？

### 用户查询的问题

1. **太模糊**："告诉我关于它的信息"（它是什么？）
2. **太口语化**："咋整啊这个"（需要转换成正式表达）
3. **太宽泛**："人工智能"（需要具体化）
4. **太复杂**：一个问题包含多个子问题
5. **缺乏上下文**："第二章讲了啥"（需要补充文档信息）

### Query 改写的三种策略

#### 1. Query 扩展（Expansion）

**目的**：增加查询的覆盖面

**方法**：
- 添加同义词
- 添加相关词
- 添加上下文

**示例**：
```
原始查询：Python 怎么读取 CSV？
扩展后：Python 怎么读取 CSV？ pandas read_csv 文件处理 数据加载
```

#### 2. Query 分解（Decomposition）

**目的**：将复杂问题拆分成简单子问题

**方法**：
- 识别问题中的多个部分
- 分别检索每个子问题
- 合并结果

**示例**：
```
原始查询：比较 RAG 和 Fine-tuning 的优缺点，什么时候用哪个？
分解后：
1. RAG 的优点和缺点是什么？
2. Fine-tuning 的优点和缺点是什么？
3. RAG 和 Fine-tuning 的适用场景分别是什么？
```

#### 3. Query 重写（Rewriting）

**目的**：改善查询的表达方式

**方法**：
- 消除歧义
- 补充上下文
- 转换成更检索友好的形式

**示例**：
```
原始查询：那个东西咋用
重写后：LangChain 的使用方法和示例代码
（假设上下文是讨论 LangChain）
```

## HyDE：假设性文档嵌入

### 什么是 HyDE？

HyDE（Hypothetical Document Embeddings）是一种"反向思维"的检索策略：

**传统方式**：Query → 找相似文档
**HyDE 方式**：Query → 生成假设答案 → 找与假设答案相似的文档

### 类比理解

想象你要找"如何做红烧肉"的菜谱：
- **传统方式**：拿着问题"如何做红烧肉"去搜索
- **HyDE 方式**：先让 LLM 写一个假的红烧肉做法，然后用这个假做法去搜索相似的真实菜谱

### HyDE 工作流程

```
1. 用户提问："什么是 RAG？"
2. LLM 生成假设答案："RAG 是 Retrieval-Augmented Generation 的缩写，它结合了检索和生成..."
3. 对假设答案做 embedding
4. 用这个 embedding 去搜索相似的真实文档
5. 返回真实文档作为上下文
```

### 为什么 HyDE 有效？

1. **语义对齐**：假设答案的语义更接近真实文档，而不是原始问题
2. **消除 gap**：问题和答案之间的语义 gap 被消除
3. **扩展覆盖**：LLM 生成的答案可能包含多种表述，增加召回率

### HyDE 的局限

1. **LLM 幻觉**：生成的假设答案可能有错误
2. **成本**：需要额外调用 LLM
3. **延迟**：增加一次 LLM 调用的延迟
4. **不适合精确查询**：对数字、日期等精确查询效果不好

## 面试加分话术

### 问题：Query 改写有什么作用？

**回答**：
"Query 改写的核心是弥补用户表达和文档内容之间的语义 gap。用户往往用口语化、模糊的方式提问，而文档通常是正式、专业的表述。通过改写，我们可以：
1. 扩展查询覆盖面（添加同义词）
2. 拆分复杂问题（子问题分解）
3. 消除歧义（补充上下文）
这能显著提升检索的召回率和准确率。"

### 问题：HyDE 是怎么工作的？

**回答**：
"HyDE 是一种'反向检索'策略。传统方式是用问题去匹配答案，HyDE 是先让 LLM 生成一个假设性的答案，然后用这个假设答案去搜索相似的真实文档。这样做的好处是：假设答案的语义更接近真实文档，能消除问题和答案之间的语义 gap。但要注意，LLM 生成的答案可能有幻觉，所以最终还是要用真实文档作为上下文。"

### 问题：Query 改写会增加多少成本？

**回答**：
"成本主要来自 LLM 调用。一次 Query 改写大约需要 100-200 tokens，用 GPT-3.5 的话成本很低。延迟大约增加 200-500ms。但这是值得的，因为一次好的改写可以避免多次无效检索。实践中，可以对简单查询跳过改写，只对复杂查询做改写，进一步优化成本。"

## 常见追问

**Q: Query 改写和 Query 扩展有什么区别？**

A: Query 扩展是 Query 改写的子集。改写包括：扩展（添加词）、重写（换表达）、分解（拆问题）、HyDE（生成假设答案）等。

**Q: 什么时候用 Query 改写？**

A: 适用场景：
- 用户查询模糊或口语化
- 检索效果不好时
- 复杂问题需要分解

不适用：
- 精确查询（如搜索特定 ID）
- 实时性要求高（增加延迟）
- 简单查询（不需要改写）

**Q: HyDE 和普通 Query 改写怎么选？**

A: 看场景：
- HyDE：适合开放式问题，如"什么是...""如何..."
- 普通改写：适合具体问题，需要精确匹配
- 可以组合使用：先改写，再 HyDE

**Q: 如何评估 Query 改写的效果？**

A: 评估方法：
1. A/B 测试：对比有无改写的检索效果
2. 离线评估：用标注数据集计算 Recall@K
3. 人工评估：抽查改写质量
