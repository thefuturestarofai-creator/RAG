# 幻觉应对策略

## 核心概念

LLM 幻觉（Hallucination）是指模型生成看似合理但实际错误或无依据的内容。

### 类比理解

想象一个学生考试：
- **有依据的回答**：根据课本内容回答问题
- **幻觉**：没有学过的内容，但编造了一个看似合理的答案

RAG 系统的优势就是"开卷考试"，通过提供参考资料来减少幻觉。

## 幻觉的类型

### 1. 事实性幻觉

生成与事实不符的内容：

```
用户："地球到月球的距离是多少？"
幻觉回答："地球到月球的距离约为 50 万公里"
正确答案："约为 38.4 万公里"
```

### 2. 忠实性幻觉

生成与上下文不符的内容：

```
上下文："Python 由 Guido van Rossum 于 1989 年开始开发"
幻觉回答："Python 由 Guido van Rossum 于 1991 年创建"
问题：年份不一致
```

### 3. 推理性幻觉

推理过程错误：

```
上下文："A 比 B 大，B 比 C 大"
幻觉回答："所以 C 比 A 大"
正确推理："所以 A 比 C 大"
```

## 幻觉应对策略

### 策略一：Prompt 约束

在 System Prompt 中明确约束 LLM：

```python
system_prompt = """你是一个知识库助手。请严格遵守以下规则：

1. 只基于提供的上下文回答问题
2. 如果上下文没有相关信息，回答"抱歉，我没有找到相关信息"
3. 不要编造或推测任何信息
4. 引用信息时，注明来源
5. 如果不确定，明确说明不确定性

上下文：
{context}

请基于以上上下文回答用户问题。"""
```

**关键点**：
- 明确要求"只基于上下文"
- 提供"不知道"的出口
- 要求引用来源
- 鼓励表达不确定性

### 策略二：来源引用

要求 LLM 在回答中引用来源：

```python
system_prompt = """请基于以下参考资料回答问题。
在回答中，使用 [1], [2] 等标记引用来源。

参考资料：
[1] 文档1内容
[2] 文档2内容
[3] 文档3内容

问题：{question}
"""
```

**示例输出**：
```
根据资料，Python 是一种高级编程语言 [1]，由 Guido van Rossum 创建 [2]。
它在数据科学领域非常流行 [3]。
```

**好处**：
- 用户可以验证信息来源
- 迫使 LLM 基于上下文回答
- 便于追溯和排查问题

### 策略三：Temperature 控制

降低 Temperature 减少随机性：

```python
response = client.chat.completions.create(
    model="gpt-3.5-turbo",
    messages=[{"role": "user", "content": prompt}],
    temperature=0,  # 最低随机性
)
```

**Temperature 对比**：
- `temperature=0`：最确定性，适合事实问答
- `temperature=0.3`：低随机性，适合大多数场景
- `temperature=0.7`：中等随机性，适合创意写作
- `temperature=1.0`：高随机性，不推荐用于 RAG

**注意**：Temperature 设为 0 并不能完全消除幻觉，但能减少随机性导致的错误。

### 策略四：Relevance 阈值过滤

只使用相关度高的文档：

```python
def filter_by_relevance(query, documents, threshold=0.7):
    """过滤掉相关度低于阈值的文档"""
    filtered = []
    for doc in documents:
        relevance = calculate_relevance(query, doc)
        if relevance >= threshold:
            filtered.append(doc)
    return filtered
```

**阈值设置**：
- 0.8+：高精度，可能遗漏
- 0.6-0.8：平衡
- 0.6 以下：高召回，可能有噪声

**动态阈值**：
```python
def dynamic_threshold(query_complexity):
    """根据问题复杂度动态调整阈值"""
    if query_complexity == "simple":
        return 0.8  # 简单问题，要求高精度
    elif query_complexity == "medium":
        return 0.6  # 中等问题，平衡
    else:
        return 0.4  # 复杂问题，允许更多上下文
```

### 策略五：后处理验证

生成回答后，验证其是否基于上下文：

```python
def verify_answer(answer, context, llm):
    """验证回答是否基于上下文"""
    prompt = f"""请判断以下回答是否完全基于提供的上下文。

上下文：{context}
回答：{answer}

判断标准：
1. 回答中的每个事实是否有上下文支持
2. 是否有编造或推测的内容
3. 数字、日期、名称是否准确

请回答"是"或"否"，并解释原因。"""

    verification = llm.generate(prompt)
    return verification
```

### 策略六：多次生成取交集

生成多次回答，取共同部分：

```python
def generate_with_consensus(question, context, llm, n=3):
    """生成多次回答，取共同部分"""
    answers = []
    for _ in range(n):
        answer = llm.generate(question, context)
        answers.append(answer)

    # 提取共同陈述
    common_statements = extract_common_statements(answers)
    return common_statements
```

**原理**：幻觉通常是随机的，多次生成的共同部分更可能是事实。

## 综合防御方案

### 推荐配置

```python
rag_config = {
    # Prompt 约束
    "system_prompt": "严格基于上下文回答，不确定时说不知道",

    # Temperature
    "temperature": 0,

    # 相关度阈值
    "relevance_threshold": 0.7,

    # 来源引用
    "enable_citation": True,

    # 后处理验证
    "enable_verification": True,
}
```

### 多层防御

```
用户查询
    ↓
[第一层] Relevance 阈值过滤 → 过滤低相关文档
    ↓
[第二层] Prompt 约束 → 约束 LLM 行为
    ↓
[第三层] Temperature=0 → 减少随机性
    ↓
[第四层] 后处理验证 → 验证回答
    ↓
最终回答
```

## 面试加分话术

### 问题：RAG 系统怎么减少幻觉？

**回答**：
"我采用多层防御策略：
1. Prompt 约束：明确要求只基于上下文回答，不确定时说不知道
2. 来源引用：要求回答中标注来源，便于验证
3. Temperature 控制：设为 0，减少随机性
4. Relevance 阈值：过滤低相关度文档，减少噪声
5. 后处理验证：生成后验证回答是否基于上下文

这些策略组合使用，可以显著减少幻觉。"

### 问题：Temperature 设为 0 还是有幻觉怎么办？

**回答**：
"Temperature 为 0 只是减少随机性，不能完全消除幻觉。更根本的方法是：
1. Prompt 约束更严格
2. 提供更相关的上下文
3. 后处理验证
4. 如果验证失败，回复'无法回答'

另外，幻觉也可能是上下文本身有误，需要检查数据源质量。"

### 问题：怎么平衡幻觉控制和回答质量？

**回答**：
"这是个权衡。过严的约束可能导致回答生硬或频繁说不知道。我的做法是：
1. 核心事实类问题：严格约束
2. 开放性问题：适度放松
3. 监控'不知道'的比例，如果太高说明检索有问题

关键是根据业务场景调整，客服系统要严格，创意助手可以放松。"

### 核心话术：RAG 评估与优化实践

**问题：你怎么做 RAG 系统的评估和优化？**

**回答**：
"我用Ragas框架对系统做了定量评估。初始版本的Faithfulness只有0.72，分析发现主要是Chunk太小导致上下文不完整。把Chunk Size从500调到800并加入章节标题前缀后，Faithfulness提升到0.89。"

**详细版**：
"在 RAG 系统评估和优化方面，我的做法是：

1. 建立评估体系
   - 使用 Ragas 框架做自动化评估
   - 覆盖四大指标：Faithfulness、Answer Relevancy、Context Precision、Context Recall
   - 构造评估数据集：从生产日志采样 + 人工标注

2. 诊断问题
   - 初始版本 Faithfulness 只有 0.72
   - 分析低分样本，发现主要原因是 Chunk Size 太小（500 tokens）
   - 小 Chunk 导致上下文不完整，LLM 需要'脑补'信息

3. 优化方案
   - 将 Chunk Size 从 500 调到 800
   - 在 Chunk 前加入章节标题作为前缀，提供上下文线索
   - 优化 Prompt，增加'只基于上下文回答'的约束

4. 优化效果
   - Faithfulness 从 0.72 提升到 0.89
   - Answer Relevancy 从 0.78 提升到 0.85
   - 用户满意度从 70% 提升到 92%

5. 持续监控
   - 每周抽样评估，跟踪指标变化
   - 建立反馈循环：评估 → 分析 → 优化 → 再评估"

## 常见追问

**Q: 幻觉能完全消除吗？**

A: 目前不能完全消除，但可以显著减少。RAG 通过提供参考资料，比纯 LLM 幻觉少很多。

**Q: 如何监控生产环境的幻觉？**

A: 
1. 定期抽样评估 Faithfulness 指标
2. 用户反馈收集（点赞/点踩）
3. 关键回答人工审核
4. 设置自动告警（低 Faithfulness）

**Q: 不同模型的幻觉程度一样吗？**

A: 不一样。通常：
- 大模型比小模型幻觉少
- 专用模型比通用模型幻觉少
- 更新的模型比旧模型幻觉少

建议用最新版本的模型，并做评估对比。

**Q: RAG vs Fine-tuning 哪个幻觉少？**

A: RAG 幻觉通常更少，因为：
- RAG 有明确的参考资料
- Fine-tuning 依赖训练数据记忆
- RAG 可以说"不知道"，Fine-tuning 倾向于给出答案
