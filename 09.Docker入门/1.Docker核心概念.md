# Docker 核心概念

## 什么是 Docker？

想象一个**集装箱**系统：在集装箱出现之前，运输货物很麻烦——散装的货物容易损坏，不同货物需要不同的运输方式。集装箱的出现解决了这个问题：把货物装进标准化的集装箱，不管里面装的是什么，都可以用同样的方式运输。

Docker 做的是同样的事情：把应用程序和它需要的一切（代码、运行时、库、配置）打包成一个标准化的"容器"，可以在任何支持 Docker 的机器上运行。

## 核心概念

### 1. 镜像（Image）

镜像是一个只读的模板，包含了运行应用所需的一切。

类比：镜像就像一个**菜谱**，里面写清楚了需要什么食材、怎么处理、怎么烹饪。

```bash
# 拉取镜像
docker pull python:3.11

# 查看本地镜像
docker images
```

### 2. 容器（Container）

容器是镜像的运行实例。一个镜像可以创建多个容器。

类比：如果镜像是"菜谱"，容器就是**照菜谱做出来的菜**。同一份菜谱可以做出很多盘相同的菜。

```bash
# 创建并运行容器
docker run -d --name my-app python:3.11

# 查看运行中的容器
docker ps

# 停止容器
docker stop my-app
```

### 3. Dockerfile

Dockerfile 是一个文本文件，描述了如何构建镜像。

类比：Dockerfile 就像"菜谱"，告诉 Docker 如何一步步制作"安装光盘"。

```dockerfile
# 基础镜像
FROM python:3.11-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件
COPY requirements.txt .

# 安装依赖
RUN pip install -r requirements.txt

# 复制应用代码
COPY . .

# 暴露端口
EXPOSE 8000

# 启动命令
CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]
```

### 4. Docker Compose

Docker Compose 用于定义和运行多容器应用。

类比：如果 Dockerfile 是"单道菜的菜谱"，Docker Compose 就是"整桌宴席的菜单"，一次可以启动多个服务。

```yaml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "8000:8000"

  database:
    image: chromadb/chroma
    ports:
      - "8001:8000"
```

### 5. Docker 与虚拟机的区别

| 对比项 | Docker 容器 | 虚拟机 |
|--------|-------------|--------|
| 启动速度 | 秒级 | 分钟级 |
| 资源占用 | 轻量（共享宿主机内核） | 重量（独立操作系统） |
| 隔离性 | 进程级隔离 | 完全隔离 |
| 镜像大小 | MB 级 | GB 级 |
| 性能 | 接近原生 | 有损耗 |

类比：
- 虚拟机：在一栋大楼里隔出独立的公寓，每套公寓有自己的水电系统
- Docker：在大楼里划分办公区，共享水电，但各有各的工位

## 在 RAG 项目中的应用场景

1. **环境一致性**：确保开发、测试、生产环境完全一致
2. **快速部署**：一键部署 RAG 应用到任何服务器
3. **微服务架构**：将 RAG 系统拆分为多个容器（API 服务、向量数据库、LLM 代理等）
4. **资源隔离**：不同服务使用不同的容器，互不影响

## 面试常问问题

### Q1: Docker 和虚拟机有什么区别？

**答**：主要区别：1）虚拟机包含完整的操作系统，Docker 容器共享宿主机内核。2）Docker 启动秒级，虚拟机分钟级。3）Docker 镜像 MB 级，虚拟机 GB 级。4）Docker 性能接近原生，虚拟机有性能损耗。5）Docker 隔离性不如虚拟机，但对于大多数应用足够。

### Q2: 什么是 Dockerfile？它有什么作用？

**答**：Dockerfile 是一个文本文件，包含了一系列指令，描述了如何构建 Docker 镜像。它的作用：1）自动化构建——一键构建镜像，无需手动操作。2）版本控制——Dockerfile 可以用 Git 管理，追踪变更。3）可复现——任何人用同一个 Dockerfile 都能构建出相同的镜像。4）文档化——Dockerfile 本身就是应用部署的文档。

### Q3: Docker Compose 适用于什么场景？

**答**：Docker Compose 适用于需要多个容器协同工作的场景。例如：1）Web 应用 + 数据库。2）前端 + 后端 + 缓存。3）RAG 系统（API 服务 + 向量数据库 + Redis）。Compose 的好处是用一个 YAML 文件定义所有服务，一条命令启动整个系统。

### Q4: 为什么用 Docker 部署 RAG 项目？

**答**：RAG 项目通常涉及多个组件（FastAPI、Chroma 向量数据库、可能还有 Redis 缓存等），每个组件有不同的依赖和配置。Docker 的价值：1）**环境一致性**——开发、测试、生产环境完全一致，避免"在我电脑上能跑"的问题；2）**一键部署**——把所有组件打包成镜像，一条命令部署到任何服务器；3）**依赖隔离**——不同组件的 Python 版本、系统库互不冲突；4）**易于扩展**——通过 `docker-compose up --scale api=3` 快速水平扩展；5）**回滚方便**——镜像有版本标签，出问题可以快速回退到上一个版本。
