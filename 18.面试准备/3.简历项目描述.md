# 简历项目描述

## 核心原则

简历项目描述就像"电梯演讲"——在有限的空间内展示你的技术能力和项目价值。

### 类比理解

想象你在电梯里遇到 HR，只有 30 秒介绍自己：
- **关键词**：让 HR 快速抓住重点
- **量化数据**：用数字证明你的能力
- **技术亮点**：展示你的技术深度

## 简历描述结构

### 推荐结构

```
项目名称 | 技术栈 | 时间
项目描述：一句话概括项目目标和价值
主要职责：
- 职责1：动作 + 结果（量化）
- 职责2：动作 + 结果（量化）
- 职责3：动作 + 结果（量化）
技术亮点：
- 亮点1
- 亮点2
```

### 示例

```
智能知识库问答系统（RAG）| Python, LangChain, Chroma, FastAPI | 2024.01-2024.06

项目描述：基于 RAG 技术构建企业内部知识库问答系统，支持多格式文档解析和智能问答，
将员工信息检索时间从平均 5 分钟降低到 30 秒。

主要职责：
- 设计并实现 RAG 核心管道，包括文档解析、分块、向量化、检索、生成五大模块
- 优化检索准确率：引入混合检索（BM25 + 向量）+ Re-ranking，准确率从 60% 提升至 85%
- 解决幻觉问题：设计多层防御策略（Prompt 约束 + 来源引用 + 后处理验证），Faithfulness 达 0.92
- 构建评估体系：基于 Ragas 框架实现自动化评估，覆盖四大核心指标

技术亮点：
- 实现 BM25 + 向量检索的 RRF 融合算法，检索准确率提升 25%
- 设计滑动窗口分块策略，长文档语义完整性保持率提升 30%
- 基于 FastAPI 构建 RESTful API，支持流式输出，并发处理能力 100+ QPS
```

## 关键词提炼

### 技术关键词

**核心概念**：
- RAG（Retrieval-Augmented Generation）
- 向量检索 / 语义检索
- 混合检索（Hybrid Search）
- 重排序（Re-ranking）
- Prompt Engineering

**技术栈**：
- LangChain / LlamaIndex
- Chroma / Milvus / Pinecone
- OpenAI API / 文心一言 / 通义千问
- FastAPI / Flask
- Docker / Kubernetes

**算法模型**：
- Embedding 模型：text-embedding-ada-002、BGE、M3E
- Re-ranking 模型：BGE-Reranker、Cross-Encoder
- LLM：GPT-3.5/4、Claude、LLaMA

### 业务关键词

**价值量化**：
- 效率提升：XX%
- 成本降低：XX%
- 准确率：XX%
- 响应时间：XX ms

**应用场景**：
- 知识库问答
- 客服机器人
- 文档检索
- 智能助手

## 量化数据技巧

### 常用量化指标

**性能指标**：
- 检索准确率：Precision@K、Recall@K、MRR
- 回答准确率：Faithfulness、Answer Relevancy
- 响应时间：P50、P95、P99
- 吞吐量：QPS（Queries Per Second）

**业务指标**：
- 效率提升：检索时间从 X 降低到 Y
- 成本节省：每年节省 X 人天
- 用户满意度：NPS、好评率

**规模指标**：
- 文档数量：X 万篇
- 向量维度：X 维
- 并发能力：X QPS

### 量化数据示例

```
不好的描述：
"优化了检索系统，提升了准确率"

好的描述：
"引入混合检索 + Re-ranking 策略，检索准确率（Precision@5）从 60% 提升至 85%，提升 42%"
```

```
不好的描述：
"处理了长文档问题"

好的描述：
"设计滑动窗口分块策略（overlap 20%），长文档（>5000字）语义完整性保持率从 70% 提升至 91%"
```

## 技术亮点提炼

### 常见技术亮点

**检索优化**：
- 混合检索（BM25 + 向量）的 RRF 融合
- Cross-Encoder Re-ranking
- Query 改写和扩展
- 多路召回策略

**生成优化**：
- Prompt Engineering（Few-shot、CoT）
- 幻觉检测和防御
- 来源引用机制
- 流式输出实现

**工程优化**：
- 向量数据库选型和优化
- 缓存策略（Embedding 缓存、结果缓存）
- 异步处理和并发控制
- Docker 容器化部署

**评估体系**：
- Ragas 自动化评估
- A/B 测试框架
- 监控告警系统
- 持续优化流程

### 亮点描述模板

```
技术亮点：[一句话概括]
- 问题：[描述问题]
- 方案：[描述方案]
- 效果：[量化结果]
```

**示例**：
```
技术亮点：实现混合检索 + Re-ranking 的两阶段检索架构
- 问题：单一向量检索准确率只有 60%，无法满足业务需求
- 方案：引入 BM25 稀疏检索，使用 RRF 算法融合，再用 Cross-Encoder 重排序
- 效果：检索准确率提升至 85%，响应时间控制在 200ms 以内
```

## 常见错误

### 错误 1：只列技术，没有成果

```
不好的描述：
"使用 LangChain 构建 RAG 系统，使用 Chroma 存储向量，使用 FastAPI 构建 API"

好的描述：
"基于 LangChain 构建 RAG 系统，检索准确率达到 85%，响应时间 < 200ms"
```

### 错误 2：描述太笼统

```
不好的描述：
"优化了系统性能"

好的描述：
"通过引入缓存和异步处理，API 响应时间从 2s 降低到 500ms"
```

### 错误 3：没有量化数据

```
不好的描述：
"提升了检索准确率"

好的描述：
"检索准确率（Precision@5）从 60% 提升至 85%，提升 42%"
```

### 错误 4：技术亮点不突出

```
不好的描述：
"实现了 RAG 系统的各种功能"

好的描述：
"实现 BM25 + 向量检索的 RRF 融合算法，解决单一检索准确率低的问题"
```

## 面试加分话术

### 如何描述项目

```
"我的 RAG 项目主要解决企业知识库问答的效率问题。
技术上，我实现了混合检索 + Re-ranking 的两阶段架构，
检索准确率达到 85%。
工程上，我建立了完整的评估体系，用 Ragas 做自动化评估。
最大的收获是深入理解了 RAG 系统的复杂性，学会了用数据驱动优化。"
```

### 如何回答"项目难点"

```
"最大的难点是检索准确率。初始版本只有 60%，我尝试了多种方法：
1. 优化 embedding 模型
2. 引入混合检索
3. 添加 Re-ranking
最终提升到 85%。这个过程让我学会了用评估驱动优化。"
```

### 如何回答"技术选型"

```
"技术选型我主要考虑三个因素：
1. 效果：BGE 在中文场景效果最好
2. 成本：Chroma 开源免费，适合原型验证
3. 生态：LangChain 社区活跃，文档完善
后续如果规模扩大，可以切换到 Milvus。"
```
