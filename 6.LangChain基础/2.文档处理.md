# 文档处理

## 为什么需要文档处理？

在 RAG 系统中，我们需要将外部知识（文档）导入系统。但大语言模型有上下文长度限制，不能把整本书一次性塞进去。所以我们需要：

1. **加载**文档
2. **切分**成小块
3. **存储**到向量数据库

类比：就像把一本厚书切成小卡片，每张卡片记录一个小知识点，方便后续查找。

## 核心概念

### 1. Document Loader（文档加载器）

LangChain 提供了多种文档加载器：

| 加载器 | 用途 | 文件格式 |
|--------|------|----------|
| `TextLoader` | 加载纯文本文件 | .txt |
| `PDFLoader` | 加载 PDF 文件 | .pdf |
| `CSVLoader` | 加载 CSV 文件 | .csv |
| `DirectoryLoader` | 加载整个目录 | 多种格式 |
| `WebBaseLoader` | 加载网页内容 | URL |

```python
from langchain_community.document_loaders import TextLoader

# 加载文本文件
loader = TextLoader("knowledge.txt", encoding="utf-8")
documents = loader.load()

# 每个 document 包含 page_content 和 metadata
print(documents[0].page_content)  # 文档内容
print(documents[0].metadata)      # 元数据（文件名、路径等）
```

### 2. Text Splitter（文本切分器）

为什么需要切分？
- LLM 的上下文窗口有限
- 太长的文档检索效果差
- 小块文档更容易找到精确答案

#### RecursiveCharacterTextSplitter（推荐）

最常用的切分器，它会按照优先级尝试不同的分隔符：

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # 每个块的最大字符数
    chunk_overlap=50,     # 相邻块的重叠字符数
    separators=["\n\n", "\n", "。", "！", "？", "，", " ", ""]
)

chunks = splitter.split_documents(documents)
```

参数说明：
- `chunk_size`：每块的最大大小（太大会超出上下文，太小会丢失语义）
- `chunk_overlap`：重叠区域（保证切分点附近的内容不丢失）

类比：Text Splitter 就像一个**智能切纸机**，它不会在句子中间切断，而是找合适的位置（句号、换行符）来切。

#### 切分策略对比

| 策略 | 优点 | 缺点 |
|------|------|------|
| 按字符数切分 | 简单 | 可能切断句子 |
| 按段落切分 | 保持语义完整 | 块大小不均匀 |
| RecursiveCharacterTextSplitter | 智能选择切分点 | 需要调参 |
| 按语义切分 | 语义最完整 | 计算成本高 |

### 3. Document 对象

切分后的每个块是一个 `Document` 对象：

```python
Document(
    page_content="文档内容文本...",    # 实际内容
    metadata={
        "source": "knowledge.txt",    # 来源文件
        "page": 1                     # 页码（PDF）
    }
)
```

## 在 RAG 项目中的应用场景

1. **知识库构建**：将企业文档、产品手册、FAQ 等加载并切分，构建知识库
2. **网页知识抓取**：加载网页内容，切分后存储到向量数据库
3. **多格式支持**：通过不同的 Loader 支持 PDF、Word、Markdown 等多种格式
4. **增量更新**：当文档更新时，重新加载和切分，更新向量数据库

## 面试常问问题

### Q1: chunk_size 和 chunk_overlap 应该怎么设置？

**答**：没有固定答案，需要根据场景调优。一般建议：
- `chunk_size`：200-1000 字符。太小会丢失上下文，太大检索不够精确
- `chunk_overlap`：chunk_size 的 10%-20%。重叠可以防止重要信息被切断
- 经验值：中文场景常用 chunk_size=500，chunk_overlap=50

### Q2: 为什么推荐 RecursiveCharacterTextSplitter？

**答**：它采用递归策略，按照优先级依次尝试 `\n\n`（段落）、`\n`（行）、句号、逗号、空格等分隔符，尽量在语义边界处切分。这样既能保证块大小一致，又能尽量保持语义完整。它是 LangChain 官方推荐的默认切分器。

### Q3: chunk_overlap 有什么作用？为什么需要重叠？

**答**：切分时，如果一个完整的句子或概念正好被切成两半，分别在两个块中，检索时可能只找到其中一个块，导致信息不完整。chunk_overlap 让相邻块有重叠区域，确保切分点附近的内容在两个块中都存在，提高了检索的召回率。

### Q4: Document 对象长什么样？metadata 有什么用？

**答**：LangChain 的 Document 对象包含两个核心字段：1）`page_content`（str）——文档的实际文本内容；2）`metadata`（dict）——元数据，通常包含 `source`（来源文件路径）、`page`（页码，PDF 时有）等信息。metadata 的作用：**来源追溯**——RAG 回答时可以告诉用户答案来自哪个文件、哪一页；**过滤检索**——可以在检索时按 metadata 过滤（如只在某个文件中搜索）；**去重管理**——利用 metadata 中的唯一标识管理文档的增删改。
