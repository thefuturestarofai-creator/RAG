# 项目规划与代码组织

> 学完了 Python 基础、进阶语法、HTTP 和 Git，现在需要把这些知识**整合起来**。本章教你如何把一个需求拆成函数和类，如何组织代码结构，以及如何用 Git 管理项目。

---

## 1. 用"盖房子"来理解项目规划

盖房子的流程：

1. **画图纸**（需求分析）→ 要盖几层、几个房间、什么功能
2. **打地基**（搭建框架）→ 确定代码结构、文件组织
3. **砌墙**（编写模块）→ 一个模块一个模块地写
4. **装修**（测试优化）→ 测试每个功能、优化代码
5. **入住**（部署使用）→ 运行项目

**写代码也是一样的流程，不能上来就写，要先规划。**

## 2. 需求分析：把需求拆成小任务

### 示例：命令行待办事项管理器

**原始需求**：做一个命令行待办事项管理器

## 3. 代码组织：文件和目录结构

### 3.1 简单项目（单文件）

对于小项目，一个文件就够了：

```
todo_app/
├── main.py              # 所有代码都在这里
├── todos.json           # 数据文件（运行时生成）
└── .gitignore
```

### 3.2 中等项目（多文件）

当代码变多时，按功能拆分文件：

```
todo_app/
├── models.py            # 数据模型（TodoItem 类）
├── manager.py           # 业务逻辑（TodoManager 类）
├── storage.py           # 数据存储（JSON 读写）
├── main.py              # 主程序入口（用户交互）
├── todos.json           # 数据文件
├── requirements.txt     # 依赖列表
├── .gitignore
└── README.md
```

### 3.3 大型项目（包结构）

```
todo_app/
├── todo/                # 包目录
│   ├── __init__.py      # 标记为 Python 包
│   ├── models.py        # 数据模型
│   ├── manager.py       # 业务逻辑
│   └── storage.py       # 数据存储
├── main.py              # 主程序入口
├── tests/               # 测试代码
│   └── test_manager.py
├── requirements.txt
├── .gitignore
└── README.md
```

## 4. 设计类和函数的原则

### 4.1 单一职责原则

每个类/函数只做**一件事**：

```python
# 好的写法：职责清晰
class TodoItem:
    """只负责表示一个待办事项的数据"""
    pass

class TodoManager:
    """只负责管理待办事项的增删改查"""
    pass

class JsonStorage:
    """只负责数据的读写"""
    pass

# 不好的写法：什么都往一个类里塞
class TodoApp:
    """既管数据，又管逻辑，还管文件读写"""
    pass
```

### 4.2 函数设计

好的函数应该：

1. **输入明确**：参数清晰
2. **输出明确**：返回值有定义
3. **功能单一**：只做一件事
4. **可测试**：不依赖外部状态（尽量）

```python
# 好的函数
def filter_completed(todos: list) -> list:
    """过滤出已完成的待办事项"""
    return [t for t in todos if t.completed]

# 不好的函数：做了太多事
def do_everything(todos):
    """过滤、排序、打印、保存... 全做了"""
    pass
```

### 4.3 类的设计

```python
class TodoItem:
    """单个待办事项"""

    def __init__(self, title, description=""):
        self.title = title
        self.description = description
        self.completed = False
        self.created_at = datetime.now().isoformat()

    def mark_completed(self):
        """标记为已完成"""
        self.completed = True

    def to_dict(self):
        """转为字典（用于保存到 JSON）"""
        return {
            "title": self.title,
            "description": self.description,
            "completed": self.completed,
            "created_at": self.created_at
        }

    @classmethod
    def from_dict(cls, data):
        """从字典创建（用于从 JSON 加载）"""
        item = cls(data["title"], data.get("description", ""))
        item.completed = data.get("completed", False)
        item.created_at = data.get("created_at", "")
        return item
```

## 5. 错误处理策略

### 5.1 分层处理

```
用户输入层    →  捕获用户输入错误，提示重新输入
业务逻辑层    →  捕获业务规则错误，返回错误信息
数据存储层    →  捕获文件读写错误，返回默认值或抛出异常
```

### 5.2 错误处理模式

```python
# 模式1：返回成功/失败
def load_data(filepath):
    try:
        with open(filepath) as f:
            return json.load(f), None    # (数据, 无错误)
    except FileNotFoundError:
        return [], "文件不存在"
    except json.JSONDecodeError:
        return [], "JSON 格式错误"

# 模式2：抛出异常
def load_data(filepath):
    with open(filepath) as f:
        return json.load(f)
    # 异常由调用者处理
```

## 6. Git 管理策略

### 6.1 提交粒度

```
# 好的提交（小而频繁）
git commit -m "添加 TodoItem 类"
git commit -m "实现 TodoManager 的增删改查"
git commit -m "添加 JSON 持久化存储"
git commit -m "实现命令行菜单交互"

# 坏的提交（大而模糊）
git commit -m "完成所有功能"
```

### 6.2 分支策略（简单项目）

```
main            ← 稳定版本
│
├── feature-todo-item      ← 开发 TodoItem 类
├── feature-manager        ← 开发 TodoManager
├── feature-storage        ← 开发 JSON 存储
└── feature-ui             ← 开发命令行界面
```

## 8. 面试官常问问题

**Q1: 如何把一个需求拆分成代码？**

A: 先识别**核心概念**（用类表示），再定义**操作**（用函数/方法表示），最后规划**用户交互**（主程序）。比如待办事项管理器：TodoItem（数据）→ TodoManager（操作）→ main（交互）。

**Q2: 什么是"单一职责原则"？**

A: 每个类/函数只做一件事。这样代码更容易理解、测试和修改。如果一个类既管数据又管UI又管文件，改任何一部分都可能影响其他部分。

**Q3: 什么时候该把代码拆成多个文件？**

A: 当一个文件超过 200-300 行，或者包含多个不相关的功能时。拆分后每个文件职责清晰，方便维护。

**Q4: Git 提交信息怎么写比较好？**

A: 用"动词+名词"格式，说明做了什么。好的例子："添加用户登录功能"、"修复文档切分的编码错误"。坏的例子："修改"、"update"、"fix"。
