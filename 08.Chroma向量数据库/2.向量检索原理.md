# 向量检索原理

## 为什么需要了解向量检索原理？

在 RAG 系统中，检索的效率和质量直接影响最终答案的质量。了解向量检索的原理，可以帮助你：
- 选择合适的检索策略
- 优化检索性能
- 理解检索结果的排序

## 核心概念

### 1. 精确最近邻 vs 近似最近邻

#### 精确最近邻（Exact NN）

最简单的方法：计算查询向量与所有文档向量的距离，返回最近的 k 个。

```
查询向量 vs 文档1 → 距离 0.1
查询向量 vs 文档2 → 距离 0.5
查询向量 vs 文档3 → 距离 0.2
→ 返回文档1和文档3
```

**问题**：当文档数量很大（如 100 万）时，计算所有距离太慢了。

#### 近似最近邻（ANN - Approximate Nearest Neighbor）

ANN 是一种"差不多正确但速度快得多"的方法。它不保证找到绝对最近的邻居，但能找到"足够近"的邻居，同时速度大大提高。

类比：找最近的餐厅——精确方法是计算城市里每家餐厅的距离；ANN 方法是只看你所在区域的餐厅，可能错过稍远一点的好餐厅，但速度快 100 倍。

### 2. HNSW 索引原理

HNSW（Hierarchical Navigable Small World）是最常用的 ANN 算法之一，Chroma 默认使用它。

#### HNSW 的类比：多层地图

想象一个城市的导航系统：

**第0层（最底层）**：详细地图，每个路口都有
- 节点：A — B — C — D — E — F — G — H

**第1层（中间层）**：简化地图，只保留主要路口
- 节点：A — C — E — G

**第2层（最顶层）**：最简化地图，只保留核心节点
- 节点：A — E

**搜索过程**：
1. 从最高层开始，快速定位到大致区域
2. 逐层下降，逐步缩小范围
3. 在最底层找到精确结果

类比：就像找人——先看他在哪个城市（顶层），再看他在哪个区（中层），最后看他在哪个小区（底层）。

#### HNSW 的优势

| 特点 | 说明 |
|------|------|
| 高查询速度 | 对数级别复杂度，百万数据毫秒级响应 |
| 高召回率 | 可以找到 95%+ 的真实最近邻 |
| 动态添加 | 可以随时添加新数据 |
| 内存友好 | 相比其他算法，内存使用更合理 |

### 3. 与传统 LIKE 查询的区别

| 对比项 | SQL LIKE 查询 | 向量检索 |
|--------|---------------|----------|
| 匹配方式 | 字面匹配 | 语义匹配 |
| 查询示例 | `WHERE content LIKE '%虚拟环境%'` | 检索"怎么设置独立的Python空间" |
| 能找到"虚拟环境"吗？ | 能 | 能（语义理解） |
| 能找到"Python独立空间"吗？ | 不能 | 能（语义相似） |
| 性能 | 全表扫描，慢 | 索引检索，快 |

### 4. 距离度量方法

向量检索需要计算两个向量的"距离"：

| 方法 | 公式 | 特点 |
|------|------|------|
| 余弦相似度 | cos(θ) | 衡量方向相似性，不受长度影响 |
| 欧氏距离 | √Σ(a-b)² | 衡量绝对距离 |
| 内积 | Σ(a×b) | 结合方向和长度 |

Chroma 默认使用余弦相似度（L2 距离归一化后等价）。

### 5. 检索质量评估

| 指标 | 说明 | 目标 |
|------|------|------|
| 召回率（Recall） | 找到的真实相关文档比例 | 越高越好 |
| 精确率（Precision） | 返回的文档中真正相关的比例 | 越高越好 |
| 查询延迟（Latency） | 每次查询所需时间 | 越低越好 |

## 在 RAG 项目中的应用场景

1. **检索优化**：理解 HNSW 原理，可以调整 ef_search 参数优化检索质量
2. **规模评估**：了解 ANN 的时间复杂度，可以评估系统能支持多大的数据量
3. **索引选择**：根据数据特点选择合适的索引类型
4. **性能调优**：在召回率和查询速度之间找到最佳平衡

## 面试常问问题

### Q1: HNSW 是如何实现高效检索的？

**答**：HNSW 构建了一个多层的图结构。底层包含所有数据点，上层是数据点的子集。搜索时从最顶层开始，利用"小世界"特性快速定位到大致区域，然后逐层下降精确查找。这种方式将搜索复杂度从 O(n) 降低到 O(log n)，百万数据可以在毫秒级完成检索。

### Q2: ANN 和精确检索有什么区别？什么时候用哪个？

**答**：精确检索遍历所有数据，保证找到最近的邻居，但速度慢（O(n)）。ANN 牺牲少量精度换取大幅速度提升（O(log n)）。选择建议：数据量小于 1 万时，精确检索足够；数据量大于 1 万时，必须用 ANN。RAG 场景通常数据量较大，一般使用 ANN。

### Q3: 如何评估向量检索的质量？

**答**：主要指标：1）召回率（Recall）——在所有真正相关的文档中，检索到了多少比例。2）精确率（Precision）——在返回的文档中，有多少是真正相关的。3）查询延迟——每次检索的时间。实际项目中，通常用 Recall@k 来评估，即返回前 k 个结果中包含多少个真正相关的文档。
