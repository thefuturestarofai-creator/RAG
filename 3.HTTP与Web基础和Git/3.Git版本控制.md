# Git 版本控制

> Git 是代码的"时光机"。它记录你每一次修改，让你可以随时回到任何一个历史版本。在 RAG 项目开发中，Git 帮你管理代码变更、多人协作、以及在出问题时快速回滚。

---

## 1. 用"时光机"来理解 Git

想象你在写一本小说：

- **没有 Git**：小说_v1.docx、小说_v2.docx、小说_最终版.docx、小说_最终版_真的最终版.docx... 一团乱麻。
- **有 Git**：自动记录每一次修改（提交），你可以随时查看"第3章在周三下午改了什么"，也可以一键回到任何版本。

### Git 的三个区域

```
工作区（Working Directory）  →  你正在编辑的文件
    │ git add
    ▼
暂存区（Staging Area）      →  准备提交的文件（打包区）
    │ git commit
    ▼
仓库（Repository）           →  已保存的历史记录（时光机）
    │ git push
    ▼
远程仓库（Remote）           →  GitHub 上的备份
```

**类比**：写信 → 把信装进信封（暂存） → 寄出去（提交） → 存到邮局服务器（推送）

## 4. 核心命令详解

### 4.1 初始化仓库

```bash
# 在项目目录下初始化 Git 仓库
cd my_rag_project
git init

# 这会创建一个 .git/ 隐藏目录，Git 的所有数据都在里面
```

### 4.2 查看状态

```bash
# 查看当前状态（最常用的命令）
git status

# 输出示例：
# On branch main
# Changes not staged for commit:
#   modified:   main.py        ← 修改了但还没暂存
# Untracked files:
#   new_file.py                ← 新文件，Git 还不认识
```

### 4.3 添加到暂存区

```bash
# 添加指定文件
git add main.py

# 添加所有修改的文件
git add .

# 添加所有 .py 文件
git add *.py
```

### 4.4 提交

```bash
# 提交暂存区的内容
git commit -m "添加了RAG检索功能"

# 提交信息要简洁明了，说明"做了什么"
# 好的提交信息: "修复文档切分时的编码错误"
# 坏的提交信息: "修改" "update" "fix bug"
```

### 4.5 查看历史

```bash
# 查看提交历史
git log

# 简洁版
git log --oneline

# 图形化展示分支
git log --oneline --graph

# 输出示例：
# a1b2c3d 添加向量检索功能
# e4f5g6h 实现文档切分
# i7j8k9l 初始化项目
```

### 4.6 查看差异

```bash
# 查看工作区和暂存区的差异
git diff

# 查看暂存区和上次提交的差异
git diff --staged

# 查看某次提交的改动
git diff a1b2c3d
```

## 5. 分支（Branch）

分支就像**平行宇宙** —— 你可以在不影响主线的情况下，尝试新功能。

### 用"树的分叉"来理解

```
main（主线）
│
├── a 初始化项目
├── b 添加文档加载
│
│   feature-search（功能分支）
│   │
│   ├── c 实现检索功能
│   ├── d 优化检索算法
│   │
│   └───┘ 合并回 main
│
├── e 合并检索功能
└── f 部署上线
```

### 分支命令

```bash
# 查看所有分支
git branch

# 创建新分支
git branch feature-search

# 切换到新分支
git checkout feature-search

# 创建并切换（上面两条命令的简写）
git checkout -b feature-search

# 在 feature-search 分支上开发...
git add .
git commit -m "实现检索功能"

# 切换回主线
git checkout main

# 合并分支
git merge feature-search

# 删除已合并的分支
git branch -d feature-search
```

## 6. 远程仓库（GitHub）

### 6.1 创建 GitHub 仓库

1. 登录 https://github.com
2. 点击右上角 "+" → "New repository"
3. 填写仓库名（如 `my-rag-project`）
4. 选择 Public 或 Private
5. 点击 "Create repository"

### 6.2 关联远程仓库

```bash
# 添加远程仓库（GitHub 给你的地址）
git remote add origin https://github.com/你的用户名/my-rag-project.git

# 查看远程仓库
git remote -v
```

### 6.3 推送到远程

```bash
# 第一次推送（-u 设置默认远程分支）
git push -u origin main

# 之后的推送
git push
```

### 6.4 从远程拉取

```bash
# 拉取远程的最新代码
git pull

# 等价于：
git fetch    # 获取远程更新
git merge    # 合并到本地
```

## 8. 常见工作流程

### 8.1 日常开发流程

```bash
# 1. 查看状态
git status

# 2. 创建功能分支
git checkout -b feature-add-reranker

# 3. 写代码...
# 编辑 main.py, retriever.py 等

# 4. 查看改了什么
git diff

# 5. 暂存和提交
git add .
git commit -m "添加 Reranker 重排序功能"

# 6. 切回主线并合并
git checkout main
git merge feature-add-reranker

# 7. 推送到 GitHub
git push
```

### 8.2 出错了怎么办

```bash
# 撤销工作区的修改（还没 add）
git checkout -- main.py

# 撤销暂存（已经 add 了，但还没 commit）
git reset HEAD main.py

# 撤销最近一次提交（保留修改）
git reset --soft HEAD~1

# 回到某个历史版本（危险！会丢失之后的修改）
git reset --hard a1b2c3d
```

## 9. 在 RAG 项目中的应用场景

| Git 操作 | RAG 中的具体用途 |
|---------|-----------------|
| `git init` | 初始化 RAG 项目 |
| `git add/commit` | 保存代码修改（如新的检索策略） |
| `git branch` | 开发新功能（如添加 Reranker）而不影响主线 |
| `git merge` | 把新功能合并到主线 |
| `git push` | 备份代码到 GitHub |
| `git pull` | 同步团队成员的代码 |
| `git log` | 查看代码变更历史 |
| `.gitignore` | 忽略 venv、.env、向量数据库数据 |
| `git reset` | 回滚有问题的代码 |

### RAG 项目的典型 Git 结构

```
my-rag-project/
├── .git/                  # Git 数据（不要手动修改）
├── .gitignore             # 忽略规则
├── .env                   # API Key（不提交！）
├── venv/                  # 虚拟环境（不提交！）
├── requirements.txt       # 依赖列表
├── main.py                # 主程序
├── retriever.py           # 检索器
├── splitter.py            # 文本切分器
├── data/                  # 知识库文档
│   ├── doc1.txt
│   └── doc2.txt
└── README.md              # 项目说明
```

## 10. 面试官常问问题

**Q1: Git 和 GitHub 有什么区别？**

A: Git 是**版本控制工具**（本地软件），GitHub 是**代码托管平台**（网站）。Git 可以独立使用（只在本地），GitHub 依赖 Git。就像 Word 和 OneDrive 的关系。

**Q2: `git add` 和 `git commit` 有什么区别？**

A: `git add` 把修改放入**暂存区**（打包准备），`git commit` 把暂存区的内容**永久记录**到仓库（存档）。暂存区让你可以选择性地提交部分修改。

**Q3: 为什么要用分支？**

A: 分支让你在**不影响主线**的情况下开发新功能。如果新功能有问题，直接删掉分支就行，主线不受影响。开发完成后合并回主线。

**Q4: `.gitignore` 有什么用？为什么要忽略 `.env` 和 `venv`？**

A: `.env` 包含 API Key 等敏感信息，泄露会导致安全问题。`venv` 文件夹很大（几百 MB），`requirements.txt` 已经记录了所有依赖，别人可以自己重建。提交这些会浪费空间和时间。

**Q5: `git pull` 和 `git fetch` 有什么区别？**

A: `git fetch` 只下载远程更新，不合并到本地（你可以先看看有什么变化）。`git pull` = `git fetch` + `git merge`，直接下载并合并。团队协作时建议先 `fetch` 再手动 `merge`，避免冲突。
