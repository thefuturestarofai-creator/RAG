# HTTP 协议

> RAG 系统的核心操作之一就是**调用 API**（比如 OpenAI 的 ChatGPT API）。而 API 的通信基础就是 HTTP 协议。理解 HTTP，你才能看懂 API 文档、调试网络问题、理解请求和响应的结构。

---

## 1. 用"餐厅点菜"来理解 HTTP

HTTP 就像在餐厅点菜：

| 餐厅场景 | HTTP 对应 | 说明 |
|---------|----------|------|
| 顾客（你） | 客户端（Client） | 发送请求的一方 |
| 餐厅厨房 | 服务端（Server） | 处理请求的一方 |
| 点菜单 | 请求（Request） | 告诉服务器你要什么 |
| 上菜 | 响应（Response） | 服务器返回的结果 |
| 菜名 | URL / 端点 | 你要访问的资源地址 |
| "要一份宫保鸡丁" | GET /menu/chicken | 请求方法 + 资源路径 |
| 菜做好了（200 OK） | 状态码 200 | 请求成功 |
| 菜卖完了（404 Not Found） | 状态码 404 | 资源不存在 |

**核心特点：HTTP 是"一问一答"的模式 —— 客户端发请求，服务器回响应，然后连接就结束了（无状态）。**

---

## 2. 请求（Request）的结构

一个 HTTP 请求包含四个部分：

```
┌─────────────────────────────────────────────┐
│ 请求行: GET /api/chat HTTP/1.1              │  ← 方法 + 路径 + 协议版本
├─────────────────────────────────────────────┤
│ 请求头:                                      │  ← 元信息
│   Host: api.openai.com                      │
│   Content-Type: application/json            │
│   Authorization: Bearer sk-xxx              │
├─────────────────────────────────────────────┤
│ 空行                                        │
├─────────────────────────────────────────────┤
│ 请求体:                                      │  ← 发送给服务器的数据
│   {"model": "gpt-3.5-turbo",               │
│    "messages": [{"role": "user",            │
│    "content": "什么是RAG？"}]}               │
└─────────────────────────────────────────────┘
```

### 2.1 请求方法（HTTP Methods）

| 方法 | 用途 | 餐厅类比 | RAG 场景 |
|------|------|---------|---------|
| `GET` | 获取数据 | 看菜单 | 查询检索结果 |
| `POST` | 提交数据 | 点菜 | 发送提问给 LLM |
| `PUT` | 更新数据 | 换菜 | 更新文档内容 |
| `DELETE` | 删除数据 | 退菜 | 删除知识库文档 |

**RAG 中最常用的是 GET 和 POST。**

### 2.2 请求头（Headers）

```python
headers = {
    "Content-Type": "application/json",           # 告诉服务器我发的是 JSON
    "Authorization": "Bearer sk-xxxxxxxxxx",      # API Key 认证
    "User-Agent": "MyRAGApp/1.0",                 # 我是谁
}
```

### 2.3 请求体（Body）

POST 请求的数据放在 Body 中，通常是 JSON 格式：

```json
{
    "model": "gpt-3.5-turbo",
    "messages": [
        {"role": "user", "content": "什么是RAG？"}
    ],
    "temperature": 0.7
}
```

## 3. 响应（Response）的结构

```
┌─────────────────────────────────────────────┐
│ 状态行: HTTP/1.1 200 OK                     │  ← 协议版本 + 状态码 + 状态描述
├─────────────────────────────────────────────┤
│ 响应头:                                      │
│   Content-Type: application/json            │
│   Content-Length: 1234                       │
├─────────────────────────────────────────────┤
│ 空行                                        │
├─────────────────────────────────────────────┤
│ 响应体:                                      │
│   {"choices": [{"message":                   │
│     {"content": "RAG是检索增强生成..."}}]}    │
└─────────────────────────────────────────────┘
```

---

## 4. 状态码（Status Code）

状态码告诉客户端请求的结果：

| 状态码 | 含义 | 餐厅类比 | RAG 场景 |
|--------|------|---------|---------|
| `200` | OK 成功 | 菜上来了 | API 调用成功 |
| `201` | Created 已创建 | 新菜已做好 | 文档创建成功 |
| `400` | Bad Request 请求有误 | 菜单上没这道菜 | 参数错误 |
| `401` | Unauthorized 未授权 | 没付钱 | API Key 错误 |
| `403` | Forbidden 禁止 | 会员专属菜品 | 没有权限 |
| `404` | Not Found 未找到 | 菜卖完了 | 资源不存在 |
| `429` | Too Many Requests 限流 | 点太多菜了 | API 调用太频繁 |
| `500` | Internal Server Error | 厨房着火了 | 服务器内部错误 |

### 状态码分类

```
1xx: 信息性（很少见）
2xx: 成功 ✅
3xx: 重定向（跳转到另一个地址）
4xx: 客户端错误 ❌（你的请求有问题）
5xx: 服务端错误 ❌（服务器出了问题）
```

## 5. JSON 数据格式

JSON（JavaScript Object Notation）是 API 通信的**通用语言**。它本质就是 Python 的字典和列表。

### JSON 与 Python 的对应关系

| JSON 类型 | Python 类型 | 示例 |
|-----------|------------|------|
| 对象 | dict | `{"name": "RAG"}` |
| 数组 | list | `[1, 2, 3]` |
| 字符串 | str | `"hello"` |
| 数字 | int/float | `42` / `3.14` |
| 布尔 | bool | `true` / `false` |
| 空值 | None | `null` |

### Python 处理 JSON

```python
import json

# Python 字典 → JSON 字符串
data = {"model": "gpt-3.5-turbo", "temperature": 0.7}
json_str = json.dumps(data, ensure_ascii=False)
print(json_str)    # '{"model": "gpt-3.5-turbo", "temperature": 0.7}'

# JSON 字符串 → Python 字典
parsed = json.loads(json_str)
print(parsed["model"])    # "gpt-3.5-turbo"
```

│       │                │   │               │
│       │                │   │               └── 查询参数

│       │                └── API 版本

## 7. 在 RAG 项目中的应用场景

| HTTP 概念 | RAG 中的具体应用 |
|----------|-----------------|
| GET 请求 | 查询向量数据库中的文档 |
| POST 请求 | 调用 OpenAI API 发送提问 |
| 状态码 200 | API 调用成功，拿到 LLM 回答 |
| 状态码 429 | API 被限流，需要等待后重试 |
| 状态码 401 | API Key 错误，需要检查配置 |
| JSON 格式 | API 请求和响应的数据格式 |
| Headers | 传递 API Key（Authorization 头） |
| URL | API 端点地址 |

### 一个完整的 API 调用流程

```
你的 RAG 程序（客户端）
    │
    │ POST https://api.openai.com/v1/chat/completions
    │ Headers: {Authorization: Bearer sk-xxx}
    │ Body: {model: "gpt-3.5", messages: [...]}
    │
    ▼
OpenAI 服务器
    │
    │ HTTP/1.1 200 OK
    │ Body: {choices: [{message: {content: "RAG是..."}}]}
    │
    ▼
你的 RAG 程序拿到回答
```

## 8. 面试官常问问题

**Q1: GET 和 POST 有什么区别？**

A: GET 用于**获取数据**，参数放在 URL 中，有长度限制，不安全（参数暴露在地址栏）。POST 用于**提交数据**，参数放在请求体中，没有长度限制，更安全。RAG 中查询用 GET，调用 LLM 用 POST。

**Q2: HTTP 和 HTTPS 有什么区别？**

A: HTTPS 是**加密**的 HTTP。数据在传输过程中被加密，防止被窃听。调用 API 时必须用 HTTPS，否则 API Key 会泄露。

**Q3: 什么是"无状态"协议？**

A: HTTP 不记住之前的请求。每次请求都是独立的，服务器不知道"你是谁"。要维持状态需要用 Cookie、Token 等机制。这就是为什么调用 API 时每次都要带上 API Key。

**Q4: 状态码 429 是什么意思？怎么处理？**

A: 429 表示**请求太频繁**（Rate Limit）。OpenAI 等 API 都有调用频率限制。处理方式：等待一段时间后重试，使用指数退避策略（1秒、2秒、4秒...）。

**Q5: JSON 是什么？为什么 API 都用 JSON？**

A: JSON 是一种轻量级的数据交换格式，本质就是键值对。它被广泛使用因为：人类可读、机器易解析、几乎所有语言都支持、比 XML 更简洁。
