# RAG 系统架构

## 一句话理解

RAG 系统就像一个"图书馆 + 专家"的组合：图书馆负责存储和查找资料（检索），专家负责阅读资料后给出回答（生成）。架构设计就是决定这个"图书馆"怎么建、"专家"怎么请、两者怎么配合。

---

## 一、整体架构图（文字描述）

=== 在线流程（用户提问）===

用户提问
  │
  ▼

└──────┬───────┘
       │
       ▼

│  检索模块     │  ← 把用户问题变成向量，去向量数据库里找最相关的文档片段

┌──────────────┐

       │

┌──────────────┐

  文档文件 → 文档处理模块 → 切分 → Embedding → 向量存储模块
```

**类比理解**：
- **文档处理模块** = 图书馆的"采购+编目"部门，负责把新书拆分、贴标签、分类上架
- **向量存储模块** = 书架，按"内容相似度"排列（不是按字母顺序，而是按语义相近程度）
- **检索模块** = 图书管理员，听到你的问题后，去书架上找最相关的几本书
- **生成模块** = 专家，翻阅管理员找来的书，写出一份通俗易懂的回答
- **API 模块** = 前台接待，负责接收问题、转交、返回结果

## 二、模块详解

### 1. 文档处理模块（Document Processing）

**职责**：把各种格式的原始文档转换成统一的、可检索的小片段。

**流程**：
```
原始文档（PDF/Word/TXT/MD）
  → 加载（Load）：读取文件内容，变成 Document 对象
  → 清洗（Clean）：去除乱码、多余空白、特殊字符
  → 切分（Split）：按段落/句子/固定长度切成小块（chunk）
  → 输出：List[Document]，每个 Document 包含 page_content 和 metadata
```

**关键设计决策**：
- 切分策略：按段落？按句子？按固定 token 数？
- chunk 大小：太大会引入噪声，太小会丢失上下文
- 重叠（overlap）：相邻 chunk 之间重叠一部分，避免在边界处切断语义

### 2. 向量存储模块（Vector Store）

**职责**：把文档片段的向量存起来，支持高效的相似度搜索。

**类比**：传统数据库像按学号查学生的档案柜，向量数据库像按"长相相似度"查人的照片墙。

**主流选择**：
| 向量数据库 | 特点 | 适用场景 |
|-----------|------|---------|
| Chroma | 轻量、嵌入式、Python 原型首选 | 学习、小项目 |
| FAISS | Facebook 开源、性能极高 | 大规模向量搜索 |
| Milvus | 分布式、企业级 | 生产环境 |
| Pinecone | 云端托管、免运维 | 不想自己搭服务 |

**存储的数据**：
- 向量（embedding）：文档片段的数值表示
- 元数据（metadata）：来源文件名、页码、切分位置等
- 原文（page_content）：原始文本内容

### 3. 检索模块（Retrieval）

**职责**：根据用户问题，从向量数据库中找到最相关的文档片段。

**检索策略**：
- **基本相似度检索**：找 top-k 个最相似的片段
- **MMR（最大边际相关性）**：在相似性和多样性之间取平衡，避免返回的片段都太雷同
- **过滤检索**：先按 metadata 过滤（如指定文件名），再做相似度搜索
- **混合检索**：关键词检索 + 向量检索 结合，取长补短

**类比**：基本检索像"找最像的5张照片"，MMR 像"找5张既像又各有特点的照片"。

### 4. 生成模块（Generation）

**职责**：把检索到的文档片段和用户问题组装成 Prompt，调用 LLM 生成回答。

**Prompt 模板示例**：
```
你是一个知识库助手。请根据以下参考资料回答用户的问题。
如果参考资料中没有相关信息，请如实说明。

参考资料：
{context}

用户问题：{question}

请回答：
```

**关键设计**：
- Prompt 工程：如何让 LLM 好好利用参考资料
- 来源标注：让 LLM 在回答中标注信息来自哪个文档
- 幻觉控制：当参考资料不足时，让 LLM 承认"我不知道"

### 5. API 模块

**职责**：对外提供 HTTP 接口，接收用户请求，协调各模块完成任务。

**典型接口设计**：
```
POST /query          # 用户提问，返回回答
POST /upload         # 上传文档，触发入库流程
GET  /health         # 健康检查
GET  /stream-query   # 流式返回回答（SSE）
```

## 三、数据流详解

### 离线数据流（文档入库）

```
1. 用户上传文档 → API 层接收文件
2. 文档处理模块加载文件 → 清洗 → 切分成 chunks
3. Embedding 模型将每个 chunk 转成向量
4. 向量 + 原文 + metadata 存入向量数据库
```

### 在线数据流（用户提问）

```
1. 用户发送问题 → API 层接收
2. Embedding 模型将问题转成向量
3. 检索模块用问题向量去向量数据库搜索 top-k 相关片段
4. 生成模块把片段 + 问题组装成 Prompt → 调用 LLM
5. LLM 返回回答 → API 层返回给用户
```

---

## 四、架构设计原则

1. **松耦合**：各模块通过接口交互，替换某个模块不影响其他模块（比如把 Chroma 换成 Milvus）
2. **可配置**：chunk 大小、top-k、Prompt 模板等都应该可配置，不需要改代码
3. **可扩展**：支持多种文档格式、多种向量数据库、多种 LLM
4. **容错性**：某个文档处理失败不影响其他文档，LLM 超时要有重试机制

## 五、面试常问问题

### Q1: RAG 系统的核心模块有哪些？
**答**：五大核心模块：文档处理模块（加载、清洗、切分）、向量存储模块（存储向量和原文）、检索模块（相似度搜索）、生成模块（Prompt 组装 + LLM 调用）、API 模块（对外接口）。

### Q2: RAG 的数据流是怎样的？
**答**：分离线和在线两条流。离线：文档→加载→切分→Embedding→向量存储。在线：用户问题→Embedding→向量检索→Prompt组装→LLM生成→返回回答。

### Q3: 如何设计一个可扩展的 RAG 架构？
**答**：关键是松耦合和接口化。每个模块定义清晰的接口（如向量数据库支持 add/search/delete），通过配置文件切换实现（如把 Chroma 换成 Milvus 只需改配置）。用依赖注入而非硬编码。

### Q4: chunk 大小怎么选？
**答**：通常 200-1000 个 token。太小会丢失上下文，太大会引入噪声。建议先用 500 token + 50 token overlap 作为起点，根据实际效果调整。不同类型的文档可能需要不同的切分策略。

### Q5: 向量数据库和传统数据库有什么区别？
**答**：传统数据库用精确匹配（WHERE name = '张三'），向量数据库用相似度搜索（找和查询向量最接近的 top-k 个向量）。传统数据库存结构化数据，向量数据库存高维向量，专门优化了近似最近邻（ANN）搜索算法。
