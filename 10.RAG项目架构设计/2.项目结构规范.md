# 项目结构规范

## 一句话理解

项目结构就像一栋楼的"户型设计"——水电管线怎么走（配置管理）、每个房间干什么（模块划分）、出了问题怎么排查（日志记录）。好的结构让项目从"能跑"变成"能维护"。

## 一、目录组织

### 类比理解

- **src/** = 主楼，所有业务逻辑都在这里
- **config/** = 配电房，集中管理所有配置
- **tests/** = 质检部门，验证每个模块是否正常
- **data/** = 仓库，存放原材料和成品
- **logs/** = 监控室，记录系统运行情况

## 二、配置管理

### 为什么不能把 API Key 写在代码里？

```python
# ❌ 错误做法：硬编码在代码里
llm = ChatOpenAI(api_key="sk-abc123...")  # 代码泄露 = Key 泄露

# ✅ 正确做法：从环境变量读取
import os
llm = ChatOpenAI(api_key=os.getenv("OPENAI_API_KEY"))
```

**原因**：
1. 代码会提交到 Git，Key 就泄露了
2. 不同环境（开发/测试/生产）的 Key 不同
3. Key 轮换时不需要改代码

### 配置管理的三层结构

```
.env 文件          → 存放实际的值（sk-xxx），不入 git
    ↓ 读取
settings.py        → 用 pydantic-settings 解析和验证
    ↓ 提供
业务代码           → from config.settings import settings
```

### .env 文件示例

```bash
# .env
OPENAI_API_KEY=sk-your-api-key-here
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_MODEL=gpt-3.5-turbo

CHROMA_PERSIST_DIR=./data/chroma_db
CHUNK_SIZE=500
CHUNK_OVERLAP=50
TOP_K=3
LOG_LEVEL=INFO
```

## 三、日志记录

### 为什么不用 print？

```python
# ❌ print 的问题
print("处理文档...")  # 没有时间、没有级别、不能输出到文件

# ✅ logging 的优势
logger.info("处理文档开始", extra={"file": "test.pdf"})
# 有时间戳、有级别、可以输出到文件、可以按级别过滤
```

### 日志级别

| 级别 | 用途 | 类比 |
|------|------|------|
| DEBUG | 调试信息，开发时用 | 施工日志（每一步都记） |
| INFO | 正常运行信息 | 工作日报（重要节点） |
| WARNING | 警告，不影响运行但需注意 | 黄灯警告 |
| ERROR | 错误，某个功能失败 | 红灯报警 |
| CRITICAL | 严重错误，系统可能崩溃 | 紧急事故 |

### 日志配置示例

```python
import logging

# 配置日志
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s: %(message)s",
    handlers=[
        logging.FileHandler("logs/app.log", encoding="utf-8"),  # 输出到文件
        logging.StreamHandler()  # 输出到控制台
    ]
)

logger = logging.getLogger(__name__)
```

## 四、错误处理策略

### 分层处理

```python
# 1. 自定义异常（语义清晰）
class DocumentLoadError(Exception):
    """文档加载失败"""
    pass

class RetrievalError(Exception):
    """检索失败"""
    pass

# 2. 在业务逻辑中抛出
def load_document(file_path: str):
    try:
        # 加载逻辑
        ...
    except FileNotFoundError:
        raise DocumentLoadError(f"文件不存在: {file_path}")

# 3. 在 API 层统一捕获
@app.exception_handler(DocumentLoadError)
async def handle_doc_error(request, exc):
    return JSONResponse(
        status_code=400,
        content={"error": str(exc)}
    )
```

### 重试策略

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),           # 最多重试 3 次
    wait=wait_exponential(min=1, max=10)  # 等待时间指数增长：1s, 2s, 4s
)
def call_llm(prompt: str) -> str:
    """调用 LLM，失败自动重试"""
    response = openai.ChatCompletion.create(...)
    return response.choices[0].message.content
```

## 五、面试常问问题

### Q1: 为什么要用 .env 文件管理配置？
**答**：三个原因：1）安全性——敏感信息（API Key）不入代码库；2）灵活性——不同环境用不同配置，不需要改代码；3）可维护性——配置集中管理，一目了然。

### Q2: 日志级别怎么选？什么时候用 ERROR？
**答**：INFO 记录正常流程（如"文档加载完成"），WARNING 记录异常但不影响整体的情况（如"某个 chunk 切分失败，已跳过"），ERROR 记录导致某个功能失败的情况（如"LLM 调用超时"），CRITICAL 记录系统崩溃的情况。

### Q3: Python 项目的标准目录结构是什么？
**答**：核心是 src 源码目录 + tests 测试目录 + config 配置目录。src 下按功能模块划分子目录（如 document_processing、vector_store、retrieval、generation），每个模块有 __init__.py。配置文件用 .env + pydantic-settings 管理。日志、数据等运行时产物单独目录，不入版本控制。

### Q4: 自定义异常有什么好处？
**答**：1）语义清晰——DocumentLoadError 比 Exception 更容易定位问题；2）分层处理——API 层可以根据异常类型返回不同的 HTTP 状态码；3）可测试——测试时可以精确断言抛出的是哪种异常。

### Q5: 什么是依赖注入？在 RAG 项目中怎么用？
**答**：依赖注入就是"不自己创建依赖，而是从外部传入"。比如 RetrievalService 不自己创建 VectorStore 实例，而是通过构造函数接收：`def __init__(self, vector_store: VectorStore)`。这样测试时可以传入 Mock 对象，切换实现时不需要修改 RetrievalService 的代码。
