# 代码重构原则

## 一句话理解

重构就像"整理房间"——东西都能用，但找起来费劲、看着也乱。重构就是把东西归类放好、扔掉不需要的、给每个抽屉贴标签，让房间整洁有序、方便使用。

---

## 一、DRY 原则（Don't Repeat Yourself）

### 什么是 DRY？

**反面教材**：
```python
# ❌ 到处重复的代码
def load_txt(file_path):
    with open(file_path, "r", encoding="utf-8") as f:
        content = f.read()
    # 清洗逻辑...
    content = content.strip()
    content = re.sub(r'\s+', ' ', content)
    return content

def load_md(file_path):
    with open(file_path, "r", encoding="utf-8") as f:
        content = f.read()
    # 清洗逻辑...（和上面一样）
    content = content.strip()
    content = re.sub(r'\s+', ' ', content)
    return content
```

**正面教材**：
```python
# ✅ 提取公共逻辑
def read_file(file_path: str) -> str:
    with open(file_path, "r", encoding="utf-8") as f:
        return f.read()

def clean_text(text: str) -> str:
    text = text.strip()
    text = re.sub(r'\s+', ' ', text)
    return text

def load_txt(file_path):
    return clean_text(read_file(file_path))

def load_md(file_path):
    return clean_text(read_file(file_path))
```

**类比**：DRY 就像"不要每间房都放一台洗衣机"——把洗衣机放在公共区域，所有房间共用。

## 二、单一职责原则（SRP）

### 什么是单一职责？

**反面教材**：
```python
# ❌ 一个函数做太多事情
def process_document(file_path):
    # 读取文件
    with open(file_path, "r") as f:
        content = f.read()
    # 清洗
    content = content.strip()
    # 切分
    chunks = [content[i:i+500] for i in range(0, len(content), 500)]
    # 向量化
    embeddings = openai.Embedding.create(input=chunks)
    # 存储
    chroma_collection.add(embeddings)
    return chunks
```

**正面教材**：
```python
# ✅ 每个函数只做一件事
def read_file(file_path: str) -> str:
    with open(file_path, "r") as f:
        return f.read()

def clean_text(text: str) -> str:
    return text.strip()

def split_text(text: str, chunk_size: int = 500) -> list:
    return [text[i:i+chunk_size] for i in range(0, len(text), chunk_size)]

def create_embeddings(texts: list) -> list:
    return openai.Embedding.create(input=texts)

def store_vectors(embeddings: list, texts: list):
    chroma_collection.add(embeddings, documents=texts)

# 组合使用
def process_document(file_path: str):
    content = read_file(file_path)
    cleaned = clean_text(text=content)
    chunks = split_text(cleaned)
    embeddings = create_embeddings(chunks)
    store_vectors(embeddings, chunks)
```

**类比**：单一职责就像"厨师分工"——切菜的只管切菜，炒菜的只管炒菜，不要让一个人又切又炒又洗碗。

## 三、模块化设计

### 好的模块化

```
rag_project/
├── src/
│   ├── document_processing/  # 文档处理模块
│   │   ├── loader.py         # 只负责加载
│   │   ├── cleaner.py        # 只负责清洗
│   │   └── splitter.py       # 只负责切分
│   │
│   ├── vector_store/         # 向量存储模块
│   │   └── store.py          # 只负责向量操作
│   │
│   ├── retrieval/            # 检索模块
│   │   └── retriever.py      # 只负责检索
│   │
│   └── generation/           # 生成模块
│       └── generator.py      # 只负责 LLM 调用
```

### 模块间通过接口交互

```python
# loader.py 的接口
class DocumentLoader:
    def load(self, file_path: str) -> List[Document]:
        ...

# splitter.py 的接口
class DocumentSplitter:
    def split(self, documents: List[Document]) -> List[Document]:
        ...

# 使用时不需要关心内部实现
loader = DocumentLoader()
splitter = DocumentSplitter()

docs = loader.load("data.txt")
chunks = splitter.split(docs)
```

## 四、代码质量检查清单

### 1. 命名规范

```python
# ❌ 不好的命名
def f(x):
    return x * 2

def proc(d, k):
    ...

# ✅ 好的命名
def calculate_double(value: int) -> int:
    return value * 2

def process_document(document: Document, chunk_size: int) -> List[Document]:
    ...
```

### 2. 类型注解

```python
# ❌ 没有类型注解
def load_document(file_path):
    ...

# ✅ 有类型注解
def load_document(file_path: str) -> List[Document]:
    ...
```

### 3. 文档字符串

```python
def split_text(text: str, chunk_size: int = 500, overlap: int = 50) -> List[str]:

    将文本切分为多个片段。

    Args:
        text: 要切分的文本

        overlap: 相邻片段的重叠字符数

    Returns:
        切分后的文本片段列表

    Raises:
        ValueError: chunk_size 必须大于 0
    """
    if chunk_size <= 0:
        raise ValueError("chunk_size 必须大于 0")
    ...
```

### 4. 错误处理

```python
# ❌ 吞掉异常
def load_document(file_path):
    try:
        ...
    except Exception:
        pass  # 不要这样!

# ✅ 正确处理异常
def load_document(file_path: str) -> List[Document]:
    try:
        ...
    except FileNotFoundError:
        raise DocumentLoadError(f"文件不存在: {file_path}")
    except Exception as e:
        logger.error(f"加载失败: {e}")
        raise
```

## 五、重构的时机

### 什么时候该重构？

1. **重复代码**：同样的逻辑出现 3 次以上
2. **函数太长**：一个函数超过 50 行
3. **参数太多**：一个函数有 5 个以上参数
4. **难以命名**：很难给函数起一个简洁的名字
5. **修改困难**：改一个地方要改很多其他地方

### 重构的步骤

```
1. 确保有测试覆盖（重构前先写测试）
2. 小步修改（每次只改一点点）
3. 每步都运行测试（确保没有破坏功能）
4. 提交代码（每完成一步就提交）
```

## 六、面试常问问题

### Q1: 什么是 DRY 原则？
**答**：DRY（Don't Repeat Yourself）是"不要重复自己"的原则。同样的代码不要写两遍，应该提取成函数或模块。好处是：修改时只需要改一处，减少出错概率，提高可维护性。

### Q2: 什么是单一职责原则？
**答**：一个函数/类只做一件事。比如 DocumentLoader 只负责加载文档，不应该同时负责清洗和切分。好处是：容易理解、容易测试、容易修改。

### Q3: 什么时候应该重构？
**答**：当出现以下信号时：1）重复代码出现 3 次以上；2）函数太长（超过 50 行）；3）修改一个地方要改很多其他地方；4）很难给函数起一个简洁的名字。

### Q4: 重构和重写的区别？
**答**：重构是在不改变功能的前提下改善代码结构，是渐进式的。重写是推倒重来，风险更大。推荐重构，因为可以逐步改善，每次改动都很小，容易验证正确性。

### Q5: 如何保证重构不破坏功能？
**答**：1）重构前先写测试；2）小步修改，每步都运行测试；3）用版本控制，每完成一步就提交；4）重构和功能开发分开，不要同时做。
